# ADR 020 — Credit ledger, pricing config, and the billing gate (WP13)

**Status:** accepted, 2026-07-04 (WP13, supervised session; owner decisions in-session and via direct migration edits)

## Context

WP13 builds ADR [006](006-auth-billing-seams.md)'s reserved seams 2–4 (credit ledger, pricing config, Stripe/iDEAL) for real, per the brief in [08-build-plan.md](../08-build-plan.md). Four pricing/scope decisions were already owner-confirmed at brief-approval time (5 signup credits, action classes 1/3/5, the `audit_answers.source_tag` column, Stripe test-mode only). This ADR records the decisions made *during* the build: the fail-closed debit rule the brief flagged as owner-required, and two rounds of pricing corrections the owner made directly in the migration files and in a new reference doc — first reversing "clarifications cost 0" (open-questions #58), then widening the whole scale ×20 for headroom (open-questions #4). **Current pricing values live in [09-pricing.md](../09-pricing.md) — this ADR covers mechanism and rationale, not the numbers, which are deliberately easy to change without a new ADR.**

**Hard boundary held throughout:** nothing in `src/answer/intent/`, `src/query/`, `src/answer/compose/`, `src/chart/`, or `src/answer/respond/types.ts` changed. Billing attaches only at the audited entry points' existing `userId` parameter and a brand-new top-level `src/billing/` module that wraps `answerQuestionAudited`/`answerClarificationReplyAudited` from the outside.

## Decisions

### 1. Debit-before-answer, with automatic compensation (owner decision, in-session)

The brief's recommendation, confirmed: reserve the credit **before** the pipeline runs (the pre-flight check + debit happens before any LLM call), and post an automatic compensating ledger entry whenever the pipeline does not deliver an answer. Rejected alternative: debit only after a confirmed answer (no reservation) — simpler ledger, but a low-balance user firing several concurrent requests could all run to completion before any of them notices the balance is gone. Debit-before closes that race: the balance is checked and reserved atomically per request via the idempotent debit (below), so a second concurrent request from the same user sees the already-reduced balance.

### 2. Two different outcomes get two different compensation treatments (open-questions #58, decided mid-WP13)

The brief's own text only worked through "internal refusal"; building it out required deciding what every non-answer outcome costs:

- **Every refusal reason**: full compensation back to 0 — no value delivered, matches ADR 015's "refusals carry no data" design exactly.
- **A thrown exception** (pipeline crash, DB blip before any response is produced): full compensation — "an unanswered question must never silently cost a credit."
- **A clarification round**: **not** free. The brief's original assumption (clarifications cost 0, same as refusals) was reversed by the owner directly in migrations 005/006 mid-session: a free doorvraag still costs real Anthropic API spend, and nothing stopped a user fishing across unrelated topics for free (ask about an unloaded topic → free clarification → try a different topic → repeat, never spending a credit). A clarification now costs its own flat **`clarification`** action-class price, deliberately kept `<=` the cheapest answer class (`simple`) so the gate never needs to charge *more* than the pre-flight debit, only ever refund the difference down to the flat price.

`src/billing/gate.ts`'s `chargeAndRun` implements this exactly: debit the estimated (`simple`) price up front; on a clarification outcome, compensate `required − clarificationPrice` (skipped only if that's `≤ 0` — a real, everyday branch today, not just a future contingency: at current prices ([09-pricing.md](../09-pricing.md), `simple` = 20, `clarification` = 10) every clarification genuinely refunds 10 credits back, netting the flat 10-credit price); on any refusal or a thrown exception, compensate the full `required` back to 0.

**A second pricing revision landed the same day, after the debit/compensation mechanism above was already built and tested:** the whole scale widened ×20 (open-questions #4) — clarification 1→10, simple 1→20, analysis 3→60, heavy 5→100, signup grant 5→100, and all four credit packs' credit amounts ×20 (a fourth pack, €250/25,000 credits, added too) — so the euro price per question is unchanged. `gate.ts`'s logic needed **no code change** for this: it already read every price from `action_class_prices` at call time rather than hardcoding a number, exactly the design ADR 006's "prices must be easy to change" called for. Only `src/billing/pricing-defaults.ts`'s data and the tests' hardcoded expected balances needed updating.

### 3. Phase 1 always estimates and prices the `simple` class

No current pipeline output can be classified `analysis`/`heavy` — those rows exist in `action_class_prices` for forward compatibility only (a future WP's drill-down or claim-verification classifier), inert until one exists. Building a classifier now for classes nothing produces would be dead code; this is stated as a scope decision, not a gap (open-questions #4).

### 4. Append-only enforced structurally, not by convention

`credit_transactions` gets a `BEFORE UPDATE OR DELETE` trigger that raises — matching this codebase's existing ethos of making invariants structural rather than "we just don't call UPDATE" (docs/05's R1/R5 pattern). A `CHECK` constraint additionally pins each `reason`'s `delta` sign (`question_cost` must be negative; `signup_grant`/`purchase`/`compensation` must be positive), so a bug can never write, say, a positive debit. Three more `CHECK`s scope `request_id`/`stripe_checkout_session_id`/`related_transaction_id` to exactly the reasons that use them.

Idempotency throughout (the debit, the compensation, the Stripe webhook credit) uses `INSERT ... ON CONFLICT ... DO NOTHING RETURNING id` against a partial unique index, branching on the empty-vs-one-row result — **never** a caught thrown unique-violation. A design-review pass caught this before it shipped: a Postgres statement error aborts the *entire* enclosing transaction even if the JS exception is caught, so a caught-and-swallowed unique violation would silently roll back anything else in the same transaction. `ON CONFLICT` never throws, so it's safe regardless of whether the caller wraps it in a transaction.

### 5. `credit_transactions.user_id` has a guarded FK to `auth.users(id)`, no cascade

The hermetic PGlite test database has no `auth` schema at all (ADR 009), so the FK is added via a `do $$ if exists (...) then execute 'alter table ... add constraint ...' end if $$` block — a safe no-op in tests, real on Supabase (mirrors migration 003's pattern for other Supabase-managed objects). Deliberately **no `on delete cascade`**: cascading a user deletion into this ledger would contradict the entire point of an append-only financial trail. The real tension this creates — GDPR erasure vs. an immutable ledger — is left open, not silently resolved, and recorded as a fresh open-question rather than guessed at.

`audit_answers.user_id` is untouched by this WP (still nullable `text`, no FK) per the brief's own words — "no migration needed for this half." It simply starts receiving real values.

### 6. The signup grant lives partly in SQL, kept tunable anyway

`public.grant_signup_credits(uuid)` (called by a thin `public.handle_new_user()` trigger on `auth.users`, both guarded the same way as the FK) runs *inside* Postgres and cannot consult `src/billing/pricing-defaults.ts` directly. It reads a `signup_grant_config` singleton row instead, which `pricing-apply.ts` keeps synced with the code constant `SIGNUP_GRANT_CREDITS` — one reviewable source of truth even though two places store the value. The grant function is idempotent by construction (`ON CONFLICT (user_id) WHERE reason = 'signup_grant' DO NOTHING`, no exception path) per Supabase's own documented warning that a *throwing* auth trigger can block real signups.

### 7. `credit_transactions.audit_answer_id` is asymmetric on purpose

Set on compensation rows (written after the pipeline's outcome, and its audit id, are known) — never on the initial debit (written *before* the pipeline call, when no audit row can exist yet). A successfully-answered question's debit therefore has no direct link to its audit row; tracing "which credit paid for which specific question" for that case is approximate (by user + timestamp proximity) rather than an exact join, a deliberately minimal Phase 1 trade-off — a tighter link is cheap to add later (thread the same client `request_id` into the audit record too) if support/dispute needs ever demand it, but isn't required by this WP's done-criteria.

### 8. Stripe: Checkout Sessions, dynamic payment methods, webhook-only fulfillment

One-time `mode: 'payment'` Checkout Sessions (never a subscription — the decided no-subscription model, docs/06-roadmap.md), `payment_method_types` deliberately omitted so Stripe's dynamic selection can surface iDEAL once enabled on the account (a Dashboard setting, not code — deferred to the live-wiring pass). Fulfillment happens **only** via the `checkout.session.completed` webhook, never the success-page redirect (the user could close the tab before it loads). The webhook's business logic (`src/billing/stripe-webhook.ts`) is framework-agnostic and hermetically tested — signature verification via `Stripe.webhooks.constructEvent` is pure crypto, no network — signed against hand-authored fixtures with `Stripe.webhooks.generateTestHeaderString`; the Next.js Route Handler is a thin adapter, mirroring how `web/app/actions.ts` is already a thin wrapper over the answer pipeline.

### 9. Supabase Auth wiring: `proxy.ts`, not `middleware.ts`

Next.js 16 renamed the `middleware.ts` file convention to `proxy.ts` (exported function `proxy`, not `middleware`) — confirmed by reading `node_modules/next/dist/docs/` directly rather than assumed from training data, per `web/AGENTS.md`'s own warning that this Next.js version has breaking changes. The proxy refreshes the Supabase session and redirects unauthenticated visits to `/login`, but per Next's own data-security guidance this is an *optimistic* check, not the authorization boundary — every Server Action that needs identity (`web/lib/current-user.ts`) verifies the session itself via `getClaims()` (JWT-validated, not a trusted cookie read), since a proxy matcher change or a Server Function moved to a different route could silently stop being covered without anyone noticing.

### 10. Adversarial review found and fixed a real concurrency bug (post-implementation, same day)

A 5-lens review (ledger-correctness, hard-boundary/anti-fabrication, security, docs-honesty, test-coverage-gaps; each finding verified by two independent skeptics who executed/read the code themselves) ran against the full diff before this WP was called done. **8 findings, 8 confirmed, 0 refuted** — the anti-fabrication and docs-honesty lenses returned zero findings (nothing in the hard boundary or the docs' measured claims was wrong); every other lens found something real:

- **[Fixed, HIGH] The debit was not actually atomic.** `chargeAndRun`'s balance check (`getBalance`) and the debit (`debitQuestion`) were two independent statements. Two concurrent requests with **different** `requestId`s (two tabs, a scripted client — the one-debit-per-`requestId` unique index only dedupes a retry of the *identical* id) from a user at exactly the required balance could both read the same pre-debit balance and both pass the check, driving the ledger negative — directly contradicting this ADR's own decision-1 text claiming the debit was "checked and reserved atomically per request." Fixed with `reserveDebit()` (`src/billing/ledger.ts`): a per-user Postgres advisory transaction lock (`pg_advisory_xact_lock(hashtext(userId))`) wrapping only the fast check-and-debit — deliberately **not** the pipeline call itself, since holding a transaction (and a pooled connection) open across a multi-second LLM response would risk exhausting the connection pool under real concurrent traffic. Proven by a new concurrency test (`tests/billing/ledger.test.ts`) that fires two concurrent debits at a user with exactly one debit's worth of balance and asserts exactly one succeeds.
- **[Fixed, MEDIUM] Two invariants were enforced only by convention or a JS-array test, not the database.** (a) Nothing tied a compensation row's `related_transaction_id` to a debit with a matching `user_id` and `reason='question_cost'` — safe today only because `gate.ts` is the sole caller and always passes matching values. (b) "`clarification` price never exceeds `simple`'s" was asserted only against the hardcoded `ACTION_CLASS_PRICES` constant, even though `docs/09-pricing.md` explicitly documents a live `UPDATE` against the pricing table as the normal way to change a price — a live edit could silently break the invariant with zero test or schema pushback, and `gate.ts` would then silently under-refund every clarification rather than erroring. Both are now real Postgres triggers, migration **008** (`credit_transactions_validate_compensation`, `action_class_prices_validate_clarification_price`), each with tests proving they reject the violating case.
- **[Fixed, MEDIUM] `web/tsconfig.json` excluded all `*.test.tsx` files from typecheck**, and the chat test's mock was untyped — together, a `GatedResponse` field-name typo (reproduced live: `balance`/`required` renamed to `creditsLeft`/`creditsNeeded`) passed both `web:typecheck` and, whenever chat.tsx didn't happen to render the drifted field, `web:test` too. Fixed by removing the exclude (plus a `types: ["@testing-library/jest-dom/vitest"]` addition so the newly-typechecked test files see jest-dom's matcher augmentation) and pinning the mock's type to the real `GatedResponse`/`ComposedResponse` types in `chat.test.tsx`.
- **[Fixed, LOW] Test-coverage gaps**: the only insufficient-balance test used a zero balance (never a nonzero-but-still-short one — `tests/billing/gate.test.ts` and `ledger.test.ts` now both cover it); the `stripe_scope`/`related_scope` CHECK constraints had no direct rejection test (added, mirroring the existing `request_id_scope` test).
- **[Confirmed, procedural, fixed] Migration 003's Data-API lockdown had never been live-verified against WP13's four new tables.** Not exploitable (the Data API is fully disabled project-wide), but unconfirmed. A live, read-only check the same day showed zero `anon`/`authenticated` grants and RLS enabled on all four — migration 003's `ALTER DEFAULT PRIVILEGES` and Supabase's RLS-auto-enable both generalized correctly, as expected but not previously *measured*. `docs/RUNBOOK.md`'s re-enable note now says to re-check per new-table-migration, not rely on the 2026-07-03 scan indefinitely.

## Alternatives considered

1. **Debit only after a confirmed answer, no reservation.** Rejected — see decision 1; a real concurrency gap.
2. **Clarifications free, same as refusals** (the brief's original assumption). Reversed by the owner mid-session (decision 2) — real API spend and an unmetered abuse vector otherwise.
3. **A continuous 1–100 complexity score per question, computed dynamically**, instead of a small fixed set of classes (the owner's own first proposal, per open-questions #58/#4). Rejected for now — no complexity classifier exists yet to make finer numbers mean anything; a plain `credits integer` column per class extends to a finer scale later without a migration once one does. The scale was still widened (1/3/5 → 10/20/60/100) precisely to leave room for that finer granularity later without another rescale.
4. **`on delete cascade` on the ledger's user FK.** Rejected — silently contradicts the append-only/audit-forever design; left as an open tension instead (decision 5).
5. **Catch thrown unique-violations for idempotency** (debit, compensation, Stripe webhook). Rejected after a design-review pass — a caught Postgres error still aborts the enclosing transaction; `ON CONFLICT ... DO NOTHING RETURNING` never throws.

## Revisit triggers

- `clarification`'s price ever moves to equal (or a misconfiguration pushes it above) `simple`'s → the partial-refund branch stops writing a compensation row (or, in the misconfigured case, silently under-refunds rather than over-charging — `gate.ts` never charges more than the pre-flight debit); re-verify against `tests/billing/gate.test.ts` before shipping such a price change.
- A future WP adds a real action-class classifier (drill-down, claim verification) → `chargeAndRun`'s "Phase 1 always estimates `simple`" comment and this ADR's decision 3 need updating together.
- An account-deletion feature is ever built → decision 5's open tension (GDPR erasure vs. append-only ledger) must be resolved, not deferred again.
- Live Stripe/Supabase-Auth account wiring (Resend SMTP, Stripe iDEAL enablement, Site URL/Redirect URLs) → RUNBOOK's live-wiring pass, deliberately deferred out of this WP by owner choice ("code first, accounts later").
