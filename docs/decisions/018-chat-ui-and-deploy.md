# ADR 018 — Chat UI architecture and first deploy

**Status:** accepted, 2026-07-04 (WP12)

## Context

Phase 0's deterministic core, LLM parsing/composition, chart spec, refusal/clarification behaviour, and audit trail are all built and passed the benchmark gate live (WP5–WP11). Nothing exists yet for a human to actually ask a question through: no web app, no deployment. ADR [001](001-single-app-vs-split.md) already fixed "one TypeScript full-stack application (Next.js)"; ADR [008](008-ui-foundation.md) already fixed the stack (Next.js App Router, TypeScript, Tailwind, Recharts) and named four binding constraints. What remained open going into this session: how the Next.js app is laid out relative to the existing `src/` backend (which runs directly under Node's native type-stripping — no build step, `erasableSyntaxOnly`, explicit `.ts`-suffixed relative imports — a scheme JSX cannot run under, per ADR [014](014-chart-spec-v1-and-renderer.md)), how it calls the two audited entry points, and how the deploy pipeline gates on the existing hermetic CI gate given this repo's direct-push-to-main workflow (CLAUDE.md, no branch protection).

## Decision

1. **One repo, `web/` as a fully independent Next.js project — NOT an npm workspace member.** This is a mid-session revision (point 9 has the full story): `web/` started as an npm workspace (`root package.json`'s `"workspaces": ["web"]`, one shared `package-lock.json`), which repeatedly fought Vercel's assumption that a deployed Next.js project's dependencies are self-contained within its own directory — traced to three separate real bugs across two failed CI deploy attempts. `web/` now has its own `package.json`, its own `package-lock.json`, and its own `node_modules`, installed independently (`npm --prefix web ci`, called out as its own CI gate step). Root `tsconfig.json`, `src/`, `tests/`, `scripts/`, and every root npm script are untouched and no longer even see `web/` as related (root `npm ci` now installs only the backend's own small dependency set again — fast, and no longer churns on frontend bumps, reversing the accepted-tradeoff language this ADR originally recorded). Root `vitest.config.ts` still needs its `exclude: [..., 'web/**']` entry regardless of workspace status — vitest's default file discovery would otherwise sweep `web/`'s own `*.test.tsx` files into the root (Node-environment) run, where they fail immediately for lacking jsdom. `web/` was scaffolded via `create-next-app` (TypeScript, Tailwind, App Router, no `src/` sub-nesting inside `web/`, own `package.json`/`tsconfig.json`/`next.config.ts`/`vitest.config.ts`). This is an internal layout choice within ADR 001's "one app, one repo" decision, not a reopening of it — "one app" was never about a shared `node_modules`.
2. **Backend code is imported directly, not duplicated, via a `web/backend` symlink to `../src` — not a bare relative path.** A throwaway spike (`web/app/page.tsx` importing `CHART_SPEC_VERSION` from `../../src/chart/types.ts`, `next build`) proved Turbopack bundles explicit `.ts`-suffixed relative imports into the root `src/` tree without issue for a plain `next build` — the constant printed correctly at prerender time, and `"allowImportingTsExtensions": true` in `web/tsconfig.json` was the only fix needed (Next's type-check step, not its bundler, was what objected). **That spike result held for local builds but not for `vercel build`** (point 9 below): Turbopack's "files outside of the project root are not resolved" restriction applies to *source-file* imports, and reaching into `../../src` was exactly that. The fix: a `web/backend` symlink to `../src`, so every backend import (`../backend/answer/audit/index.ts` etc.) stays lexically inside `web/`'s own directory tree — Turbopack's project root never needs to extend beyond `web/` for source resolution at all. No path alias was used; the symlink keeps import syntax as plain relative paths, consistent with the rest of the codebase. This is independent of, and survived, the point-1 workspace-to-independent-project revision — the symlink still lets `web/` reach backend *source* with zero package-level coupling, regardless of how `web/`'s own npm dependencies are installed. **The symlink alone was still not sufficient under `vercel build`'s own sandbox** (a stricter boundary than plain `next build`, which is why nothing caught this locally) — it additionally needed the Vercel project's `sourceFilesOutsideRootDirectory: true` setting (point 9) before the build would follow it at all.
3. **Server Actions, not a Route Handler**, call the two audited entry points (`web/app/actions.ts`, thin marshaling-only wrapper). `AuditedResponse`/`ComposedResponse` are plain-serializable (no Date/Map/class instances), so RSC serialization is a non-issue; the audited entry points already fail closed internally (a failing audit write degrades to an internal refusal rather than throwing), so the action's own error surface is just infra failures (DB/LLM unreachable). Confirmed against the installed Next 16 docs: Server Actions may be invoked directly from a client event handler (not only `<form action>`), dispatch sequentially per client (a free double-submit guard at the framework level, on top of a client-side in-flight disable), and carry no caching/revalidation semantics we need here — this is effectively a typed RPC call, not a page mutation. If real stage-status streaming is ever built (see point 5), that specifically wants a Route Handler returning a stream — named here as the future swap point, confined to one file today.
4. **DB pooling reuses `connectFromEnv()` unchanged** (`web/lib/db.ts`, module-scope singleton, `globalThis`-cached to survive dev HMR). Same code path the CLI scripts and CI already exercise. `max: 4` and the session-mode pooler (needed by `ingest`'s temp tables, not by the web app) stay as-is rather than forking a second pool config — pool exhaustion under concurrent bursts is an accepted, named risk at Phase 0 traffic levels, not a silent gap. Reference date is a real `new Date()` at request time (the one legitimate un-pinned clock in the codebase); LLM clients are fresh `new AnthropicLlmClient()` per call for both roles, matching existing live-script wiring exactly. The route/action runs on the Node.js runtime (`export const runtime = 'nodejs'` set explicitly on the page, even though `'nodejs'` is already Next 16's default — cheap insurance against a future default flip or accidental edge config) since `pg` and the pinned-CA filesystem read need real Node, not Edge.
5. **Non-streaming answer UX — a recorded, deliberate deviation from ADR 008 constraint 2's "streams stage-status" half.** A static spinner with non-claiming Dutch copy ("Bezig met het doorzoeken van CBS-cijfers…"), not real multi-stage progress events. Genuine stage-status streaming would require instrumenting the audited pipeline's internals (parse → query → compose) with progress callbacks — a pipeline change disproportionate to a "minimal, ugly is acceptable" UI (docs/03), and risking exactly the kind of pipeline-touching change CLAUDE.md asks invariant scrutiny for. Measured live latency (median 6.5s, max ~14s, WP11) is tolerable behind a spinner. Named here as the explicit future path if latency regresses. **"Regenerate"** (ADR 008's other mention in the same constraint — re-phrasing over the same validated result, writing a new audit record) is likewise **not built in WP12**; a future session should not assume it shipped here.
6. **Recharts wrapper** (`web/components/chart.tsx`, client component) over the existing `ChartSpec` — the deferred obligation ADR 014 named. Dispatches on `kind` ('line'/'bar'); rows sorted chronologically by `periodCode`, not `periodLabel`/insertion order (matching the fix the SVG renderer's own pre-push adversarial review already made, for the same reason — series can carry disjoint period sets); null values render as natural gaps (`connectNulls={false}`); provisional points get a distinct marker; every displayed numeric string is a point's `formattedValue`, never Recharts' own formatting of the raw `value` — enforced via a custom tooltip reading a sibling `_display` field per row, with every tooltip value node additionally carrying `data-label-for="<resultId>"` so each displayed string is **bound** to its source cell, and test-pinned twice: a membership check (every numeric token in the rendered DOM occurs verbatim in the spec's own strings) and a binding check (each display string renders inside the node carrying *that point's* resultId — a swap fails even though both strings would pass membership). `attributionLine`/`definitionLine`/`provisionalNote`/`nullNotes` render as plain text around the chart. **Correction (2026-07-04, post-session adversarial review):** as originally written this decision claimed a DOM-level membership test that did not exist, and the wrapper dropped `resultId` entirely — the exact membership-without-binding gap the WP8 review had already fixed once in the SVG renderer, recurred in new code. Both confirmed by two executing skeptics and fixed + test-pinned the same day.
7. **CI-gated deploy, not Vercel's git integration.** This repo pushes directly to `main` with no PRs and no branch protection (CLAUDE.md, deliberate). Vercel's GitHub App would auto-deploy every push regardless of CI outcome — that is not deploy-blocking, it just races CI. So the Vercel project is created with **no git integration connected** (confirmed once in the dashboard's Git tab), and a `deploy` job was added to `.github/workflows/ci.yml` (`needs: gate`, `if: github.ref == 'refs/heads/main'`) running `vercel pull && vercel build && vercel deploy --prebuilt --prod` against `VERCEL_TOKEN`/`VERCEL_ORG_ID`/`VERCEL_PROJECT_ID` repo secrets. A red `gate` job structurally prevents `deploy` from running via `needs:` — real deploy-blocking-on-red that works with this repo's actual workflow instead of fighting it with unusable branch protection.
8. **The pinned CA cert footgun (point 8 of the original plan) turned out worse than anticipated, and was fixed differently than planned.** `src/db/client.ts` originally read `config/supabase-prod-ca-2021.pem` via `readFileSync(new URL('./relative', import.meta.url))`. Under Turbopack's bundled Node.js runtime this failed in **two** distinct ways, found only by actually exercising the built output (`next build && next start`), never in `next dev` or a hermetic test: first, the constructed `URL` instance failed Node's internal `instanceof` brand check inside `fileURLToPath`/`readFileSync(URL)` — a cross-realm artifact of the bundled runtime, not present when the module runs unbundled (CLI scripts, CI); second, and more fundamentally, Turbopack statically recognizes the literal `new URL('./relative', import.meta.url)` pattern as a **client-asset reference** (the same convention used for bundling images/fonts) and rewrites it at build time to a `/_next/static/media/...` public path — never a real server-side filesystem path, so `outputFileTracingIncludes` (the originally planned fix) was solving the wrong problem: the file *was* traced and shipped, but the code was never going to be able to find it there. Since the CA cert is Supabase's **public** root CA (already committed to the repo, not a secret), the fix sidesteps filesystem resolution in the bundled runtime entirely: `web/next.config.ts` reads the `.pem` file at Next's *build* time (plain Node, unbundled — `next.config.ts` is loaded directly by the Next CLI, never passed through Turbopack) and bakes its contents into `DATABASE_CA_CERT` via Next's `env` config key. `src/db/client.ts` prefers `process.env.DATABASE_CA_CERT` when set, falling back to the original file read for unbundled contexts (CLI scripts, where the original pattern has always worked correctly). No Vercel dashboard secret is needed for this value, and it stays in sync automatically on every build.
9. **Six real CI runs — five failures — before the pipeline went green; the decisive fix was a Vercel project setting, not a Next.js/Turbopack config.** *(Heading corrected by the 2026-07-04 post-session review: it originally said "three attempts" — written at attempt three and never updated as attempts four and five were appended below.)* Each fix below, tested where possible by directly reproducing the CI failure against the real linked project, was necessary but not sufficient on its own:
   - **Attempt 1: "couldn't find the Next.js package... from the project directory."** Never reproduced by a plain `next build` locally. Two config fixes tried and rejected: `turbopack.root`+`outputFileTracingRoot` both pointed at the monorepo root fixed Turbopack's own resolution but broke Vercel CLI's separate output-packaging step with a doubled-path `ENOENT .../web/web/.next/package.json`; the Vercel project's `rootDirectory` set to `"web"` **while still invoking every command with `--cwd web`** double-applied the subdirectory the same way, failing even earlier with `ENOENT .../web/web/package.json` — an important detail later corrected (see attempt 3).
   - **Attempt 2: the point-2 symlink alone, reasoning that Turbopack's "outside project root" restriction only concerns source-file imports, not `node_modules` package resolution.** Fixed the local build but **not** the CI failure — pushed as its own commit, it hit the *identical* attempt-1 error again, on the bare `./app` entry point, before ever reaching an import statement the symlink touched. That disproved the theory for this specific failure: the error was never about `../../src`. The actual, documented cause: Turbopack cannot resolve `next` itself from `web/app` when `next` is hoisted to a *parent* directory's `node_modules` — an acknowledged Turbopack/npm-workspaces limitation. Fixed by making `web/` a fully independent npm project (point 1) — its own `package-lock.json`/`node_modules`, so `next` is always physically present where Turbopack looks. A parallel dead end, `install-strategy=nested` (un-hoisting *within* a still-shared workspace), fixed this symptom too but broke `vitest`'s resolution of `web`-only devDependencies (`jsdom`) — npm's workspace bin-symlinking places every workspace's binaries at the *root* `node_modules/.bin` regardless of install strategy, so `npm run test --workspace web` silently ran the root's own separately-nested, jsdom-less `vitest`. Splitting into two independent projects sidesteps this too.
   - **Attempt 3: with `web/` now independent, the "next not found" error was gone — replaced by `Symlink [project]/backend/... is invalid, it points out of the filesystem root`.** This is the real, root explanation for *all three* attempts: `vercel build` runs Next's build inside its own sandbox that refuses to resolve **anything** — package or symlinked source file — outside the directory it considers the project root, a stricter boundary than plain `next build` enforces (which is why nothing reproduced locally until `vercel build` itself was run). The actual fix is a **Vercel project setting**, documented for exactly this monorepo shape, with no `next.config.ts` or CLI equivalent: `rootDirectory: "web"` plus `sourceFilesOutsideRootDirectory: true` (Vercel's "Include source files outside of the Root Directory in the Build Step" toggle — a project-level API field, `PATCH /v9/projects/{id}`, no CLI command exists for it). With both set, `vercel` commands run from the **repo root** (not `--cwd web` — combining the two double-applies the subdirectory, per attempt 1's second failure), letting the project's own settings do the scoping instead. The local `.vercel/project.json` link file moved from `web/.vercel/` to the repo root to match.
   - **Attempt 4: with the symlink now genuinely followed, Turbopack correctly resolved the real backend source files — and then failed on `Module not found: Can't resolve 'zod'`** from inside `src/chart/schema.ts`/`src/answer/intent/schema.ts`, reached via the symlink. This was never a bug: `web/`'s independent install (attempt 2) has its own `node_modules` with no `zod`/`pg`/`@anthropic-ai/sdk` in it, because `web/package.json` never declared them — the shared-workspace layout papered over this by accident (root's hoisted copies were reachable). Splitting the lockfiles correctly surfaced a real, previously-implicit dependency: `web/`'s Server Action genuinely runtime-depends on backend code that needs these three packages. Fixed by adding `zod`, `pg`, `@anthropic-ai/sdk` (and `@types/pg`) directly to `web/package.json`, matching root's versions — not a workaround, just making an already-true dependency explicit.
   - **Attempt 5: the exact same three packages still unresolvable, despite now being declared in `web/package.json` and genuinely installed into `web/node_modules`.** The missing piece: `web/backend/...` is a *symlink* — Node/Turpoback resolve a symlinked file's own `import` statements relative to its **real, resolved path** (`<repo root>/src/...`), not the symlink's apparent location inside `web/`. Walking up from the real path never reaches `web/node_modules` at all; it reaches `<repo root>/node_modules` instead — which this session's own deploy job had stopped installing (removed as "redundant" once `web/` became independent, attempt 2). Root's `package.json` already declares the same three packages (it always has — `src/` is the original backend), so the fix was simply restoring the root `npm ci` step in the `deploy` job, previously removed on a wrong assumption that `vercel build`'s own `web/`-scoped install was the only one needed.

   Full account, including the operational mistakes made while debugging directly against the live Vercel project (an accidental duplicate project; extracting the CLI's stored auth token to call the API directly, twice, the second time with explicit owner authorization), in [lessons-learned.md](../lessons-learned.md).
10. **Dependabot gets a second entry, `/web`.** Reverses this ADR's original reasoning (a single root entry sufficed under one shared workspace lockfile) — now that `web/` has its own independent `package-lock.json` (point 1), the root entry's npm ecosystem scan does not cover it at all.
11. **Post-review hardening (2026-07-04, owner-requested adversarial review — scoreboard in [STATUS.md](../STATUS.md)).** Fixes landed from confirmed findings, beyond the corrections noted inline above: the Recharts wrapper carries `resultId` per point and binds every tooltip value node to its source cell via `data-label-for`, with binding + membership tests added (the two HIGH findings); `referenceDate` is computed in Europe/Amsterdam, not UTC (a plain UTC date is still yesterday for up to two hours after Dutch midnight, skewing "vorige maand"-style resolution); Server Action infra failures are `console.error`-logged so Vercel function logs give the owner prod visibility (the client still gets Next's masked generic error); an input-length guard (client `maxLength` 500, server-side 2000-char throw) bounds single-request token spend on the public endpoint — an infra guard, **not** rate limiting, which stays Phase 1–2 per docs/03 non-goals and ADR [005](005-caching-strategy.md); `loadCaCert` fails closed (an empty/garbled `DATABASE_CA_CERT` falls back to the committed pinned CA instead of silently weakening TLS); `.claude/launch.json` was fixed off the abandoned `--workspace` invocation; CI's npm cache key now covers both lockfiles; `buildRows`'s period label is first-wins across series with disjoint period sets.

## Alternatives considered

1. **Separate repo / split deploy for the frontend.** Rejected — reopens ADR 001 for no functional gain; doubles CI, secrets, and the mental model a future AI session must reconstruct.
2. **Next.js files placed directly under root `src/`.** Rejected — collides with the root tsconfig's `erasableSyntaxOnly`/node-type-stripping scheme (confirmed: JSX requires codegen, not just type erasure, so it cannot run under that scheme), and would blur the existing `ingestion/`/`cbs-adapter/`/`query/`/`validation/`/`answer/`/`chart/` module-boundary contract (ADR 001) with UI routes.
3. **Route Handler instead of Server Actions.** Considered and still available as the future streaming path (point 3); rejected for the initial non-streaming call because it adds a second code path (fetch client-side, JSON contract) for no benefit over a directly-awaited, plainly-serializable Server Action.
4. **Real stage-status streaming in WP12.** Rejected for now (point 5) — the honest middle ground considered (a Route Handler emitting 2–3 timing-heuristic, not pipeline-real, milestone events) was rejected too: fake progress text in a product whose entire premise is "never guess" is a small honesty smell, and it's not actually cheaper than the plain spinner once the extra Route Handler + client stream-reading code is counted.
5. **Vercel's native git integration + branch protection for deploy-blocking.** Rejected — branch protection cannot gate direct pushes to `main` under this repo's no-PR workflow, so it would not actually block a red deploy; the CI-job `needs:` dependency does.
6. **Transaction-mode Supabase pooler for the web app's DB connection.** Considered as a lighter-weight fit for a bursty serverless request pattern than the session-mode pooler `connectFromEnv()` uses (session mode exists for `ingest`'s temp tables). Deferred, not silently unconsidered: forking the pool config adds a second code path to audit for a Phase 0 traffic level where it doesn't yet matter.
7. **`turbopack.root`/`outputFileTracingRoot` config overrides, the Vercel project's `rootDirectory` setting, and `install-strategy=nested` within a still-shared workspace — four attempts to fix the CI-only `vercel build` failure while staying an npm workspace.** All four tried and rejected — see point 9. Splitting `web/` into a fully independent project needed no Vercel project configuration, no Turbopack config overrides, and no npm install-strategy tuning at all; it fixes the root cause (dependency hoisting) directly rather than working around each symptom individually.

## Consequences

- The chat UI calls only `answerQuestionAudited`/`answerClarificationReplyAudited` — no other path into the pipeline exists from `web/`.
- A future session adding real stage-status streaming or "Regenerate" has a named, scoped starting point (points 3 and 5) rather than having to rediscover the deviation from ADR 008.
- Root backend code, tests, and CI are provably unaffected by the web app's existence (measured: 445/445 tests, clean typecheck, with `web/` present).
- Two deploy-only risks were resolved before first deploy rather than discovered in prod: the `.pem` file's bundling and the Node-runtime requirement for `pg`/filesystem access.
- A third deploy-only risk (point 9) was *not* caught before the first two CI pushes — the monorepo root-resolution failure only reproduces under `vercel build` in a genuinely fresh checkout, which local `next build`/`next dev` never triggered, and it took two failed attempts (not one) to reach the actual fix, because the first fix's diagnosis (source-file resolution) was plausible, partially right about the general Turbopack restriction, and still wrong about this specific failure's cause. `next build && next start` (added after the `.pem` lesson) was not sufficient; `vercel build` itself is now the closer proxy worth reaching for when a change touches cross-directory imports or dependency layout — see the lessons-learned entry for the authorization boundary on actually running it.
- Root `npm ci` is now fast again (backend deps only, ~77 packages) — the "slower install, lockfile churns on frontend bumps" tradeoff this ADR originally accepted no longer applies; it was specific to the (since-abandoned) shared-workspace layout.
- Two lockfiles now exist (`package-lock.json`, `web/package-lock.json`) and two Dependabot entries track them — marginally more surface than the one-lockfile ideal this ADR originally aimed for, accepted because the alternative (fighting Vercel's hoisting assumptions indefinitely) proved more expensive in practice than in theory.

## Revisit triggers

- Median live latency regresses meaningfully past WP11's baseline → build real stage-status streaming via the named Route Handler path.
- Concurrent Phase 1 users show DB pool queuing → revisit the transaction-mode pooler question for the web app specifically.
- A genuine need to share more than backend *source* between root and `web/` (e.g. a shared npm package, not just the `web/backend` symlink) → revisit whether npm workspaces are worth reattempting, now with the specific Vercel/Turbopack failure mode on record.
