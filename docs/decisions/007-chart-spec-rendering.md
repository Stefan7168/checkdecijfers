# ADR 007 — Charts: server-built declarative spec, dumb client renderer

**Status:** accepted, 2026-07-02

## Context

The notes demand the visualization layer be "dom en voorspelbaar" (dumb and predictable) — charts must never re-interpret data. They also contain a large future ambition (the "Visualisatie Studio": social-media formats 1:1/4:5/9:16/16:9, custom pixel sizes, interactive charts, iframe embeds, enterprise huisstijl theming, OpenGraph images for shareable answer pages). Stefan explicitly asked (interview Q1) that these future builds be preserved and their architecture implications kept in mind. Charts are in Phase 0 scope in their simplest form.

## Decision

The pipeline's answer composer emits a **declarative chart spec** (JSON): chart type (line/bar/table only in Phase 0), series data (verbatim from validated query results), axis labels, units, source attribution, sync date. The client renders this spec with a **dumb renderer** — a thin wrapper around one charting library (Recharts, Chart.js, or ECharts; picked at implementation for bundle size/DX, swappable behind the wrapper).

Rules:

- The renderer receives only the spec; it may not fetch, compute, aggregate, round, or omit data points.
- The spec is generated by deterministic code from validated results — the LLM never produces chart data (it may at most *suggest* a chart type in the intent, which validation can override).
- Attribution and sync date are part of the spec, so no rendering path can drop them.

## Why this is the seam for the Visualisatie Studio

Every future output format is *another renderer over the same spec*: a static-image renderer (social formats, custom sizes, OpenGraph images for shareable answer pages), an embed renderer (iframe), a themed renderer (enterprise huisstijl = a theme object applied to the spec). None of them require touching the data pipeline. This is the concrete "architecture choice kept in mind" Stefan asked for.

## Alternatives considered

1. **LLM generates chart JSON directly** (the notes' early flow: "Claude returns chart-ready JSON"). Violates principle (a) — a model emitting the numbers that get plotted is exactly the hallucination surface we exist to remove. Rejected.
2. **Client-side charting from raw query results** (frontend decides what to plot). Pushes interpretation into the least testable layer and makes attribution optional. Rejected.
3. **Server-side image rendering from day one.** Needed later for exports/OpenGraph anyway, but heavier (headless rendering) and Phase 0 has no sharing features. Deferred to Phase 2 with the spec as the ready seam.

## Consequences

- Chart correctness is testable without a browser: assert spec contents against query results (invariant R6 in [05-data-rules.md](../05-data-rules.md)).
- Interactivity in Phase 0 is minimal by design; hover/zoom arrive with the Studio, not before.

## Revisit triggers

- Phase 2 sharing features → build the static-image renderer over the spec.
- Enterprise tier (Phase 3) → theme object / huisstijl support in the spec schema.
- Chosen chart library obstructs the spec model → swap behind the wrapper (that's what it's for).
