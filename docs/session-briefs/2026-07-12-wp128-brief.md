## What WP128 is

The owner's #128 request, verbatim scope: **two buttons (👍/👎) on every chat ANSWER; a
thumbs-down opens a small free-text ask; one additive write-only `answer_feedback` table keyed
on the audit row.** It collects the signal the later answer-quality lane (WP26) optimizes
against: 👎-rows become the experience-audit shortlist. Zero pipeline/LLM/prompt impact —
display + a write-only store; the R8 audit row is untouched (feedback POINTS at it, never
modifies it).

## Design decisions (the review's attack surface)

**F1 — Table (migrations/017_answer_feedback.sql, FILE ONLY; applied in the supervised window
together with the pending 016):**

```sql
create table answer_feedback (
  id bigint generated always as identity primary key,
  audit_answer_id bigint not null references audit_answers(id),
  user_id text not null,
  verdict text not null check (verdict in ('up', 'down')),
  feedback_text text,
  created_at timestamptz not null default now(),
  constraint answer_feedback_one_per_user_per_answer unique (audit_answer_id, user_id)
);
create index answer_feedback_by_user on answer_feedback (user_id);
```

House header per 015/016 style. NO explicit lockdown SQL (the 003 mechanism covers it — state
this in the header with the 011/012 precedent). The unique constraint + upsert (F2) means a
user can CHANGE their verdict (👍→👎 with text) — last write wins, `created_at` refreshed.
FK deliberately has no ON DELETE clause: nothing deletes audit rows (redaction-only), and
feedback rows themselves are hard-deleted by the GDPR paths (F3).

**F2 — Store module (`src/answer/audit/feedback.ts`, new; write-only):**
`upsertAnswerFeedback(db, { auditAnswerId, userId, verdict, feedbackText })` —
parameterized insert `on conflict (audit_answer_id, user_id) do update set verdict, feedback_text,
created_at = now()`; **ownership + kind + source guard IN the same statement**: the insert
selects from `audit_answers where id = $1 and user_id = $2 and kind = 'answer' and
source_tag = 'user'` ⟨K:gdpr-1 — the source_tag clause makes "benchmark rows can never carry
purge-exempt feedback" structural, not an accident of runner scripts⟩ so feedback can only ever
attach to the caller's OWN, USER-tagged ANSWER rows. The upsert's conflict path is only
reachable FROM a guarded row, so the guard holds on both paths. ⟨K:migration-deploy-1⟩ The
statement ends `returning id` — the Db interface exposes only `rows`, not rowCount, so the
guard's "zero rows touched" signal IS the empty rows array → return `false` (soft).
`feedback_text` trimmed, empty → null, capped at 2,000 chars (validated in the action AND
enforced here). Never CALLS a billing function ⟨K:auth-money-4 — a call-level guarantee;
actions.ts as a FILE already imports billing for the ask flows⟩.

**F3 — GDPR interplay (extend `src/answer/audit/retention.ts` + docs/05):**
- `deleteUserQuestionHistory`: additionally `delete from answer_feedback where user_id = $1`
  (hard delete — "wis volledig", and no FK references answer_feedback).
- The 2-year purge: additionally hard-delete feedback rows whose audit row falls in the purge
  window (`audit_answer_id in (select id from audit_answers where source_tag='user' and
  created_at < cutoff)` — the cutoff = the SAME `twoYearsBefore` value the redaction uses).
- ⟨K:auth-money-3/gdpr-2/migration-deploy-3, three-lens convergence⟩ **Both new deletes run
  INSIDE the existing `db.withTransaction` block of the function they extend, feedback-delete
  FIRST, redaction after** — a crash between the two can never leave feedback text behind a
  redacted row; both paths stay idempotent.
- The ledger stays structurally untouched (no billing import — existing test discipline
  extends).
- docs/05-data-rules.md GDPR section gains the answer_feedback lines in the same change.

**F4 — Server action (`web/app/actions.ts` — `submitAnswerFeedback`):**
`(auditId: number, verdict: 'up' | 'down', feedbackText?: string) → Promise<{ ok: boolean }>`.
Auth inside the action (the house `currentUserId()` gate); validates verdict ∈ {up,down} (input
is attacker-controlled — treat as `unknown`), auditId a finite positive integer, text ≤ 2,000
chars; calls F2. **FAIL-SOFT EVERYWHERE — ⟨K:auth-money-1⟩ explicitly UNLIKE the cited
deleteMyQuestionHistory precedent (which throws on unauth): this action's ENTIRE body sits in
one try/catch and every path — unauthenticated, validation failure, db error incl. the missing
table in the pre-migration window, ownership miss — returns `{ ok: false }`, never throws.**
Free feature: NO billing gate, NO new charged entry point; the function body calls no billing
function. Residual (review-accepted): no rate limit — the unique constraint bounds feedback
cardinality to the caller's own answers; request-level throttling is platform-lane, not v1.

**F5 — UI (`web/components/chat.tsx` + a new `web/components/feedback-buttons.tsx`):**
⟨K:r8-pipeline-1/ui-tests-1, two-lens convergence — the anchor reaches the ACTION result but
NOT today's render path⟩ **Work item, client-side only: `ChatMessage` gains
`auditId: number | null`, populated at message-push time from the gated result in BOTH answer
paths (`handleSubmit` and the clarification-reply flow).** Nothing R8-stored changes — this is
transient client state. Then: on `kind === 'answer'` messages with a numeric `auditId`, render
⟨K:ui-tests-5⟩ a self-contained **`FeedbackButtons` child component** (the delete-history-button
precedent: own state, receives `auditId` + the action as prop-injectable seam) after the
attribution chip. Two small ghost buttons (WP29 chip idiom), aria-labels "Nuttig antwoord" /
"Niet nuttig", ⟨K:ui-tests-3⟩ **`aria-pressed` reflecting the chosen verdict** (toggle
semantics — the verdict is changeable). 👍 → submit immediately. 👎 → open the INLINE panel
(textarea, placeholder "Wat kon beter? (optioneel)", maxLength 2000, buttons
⟨K:ui-tests-2⟩ **"Verstuur feedback"** (the bare "Verstuur" name is taken by the chat submit in
the same DOM) / "Overslaan" — Overslaan submits the bare 👎). After `{ ok: true }`: muted
"Bedankt voor je feedback.", chosen button filled, other button stays clickable (verdict change
→ upsert). `{ ok: false }`: muted "Feedback kon niet worden opgeslagen." — and the busy flag
RESETS so retry stays possible; the answer display is never affected. Double-click protected
(the component's own busy flag). Refusals, clarifications, info messages and answers with
`auditId: null` get NO buttons.

**F6 — Tests (all hermetic):**
- `tests/db/migration-017.test.ts`: table exists post-migrations; verdict CHECK enforced;
  unique (audit_answer_id, user_id) enforced; FK to audit_answers enforced; upsert path changes

- `tests/audit/feedback.test.ts` (PGlite): upsert happy path; ownership pins (another user's
  audit row → false, zero rows); kind pin (refusal/clarification row → false); text
  trim/cap/null; **retention interplay**: deleteUserQuestionHistory removes the caller's
  feedback and ONLY the caller's; the purge removes aged-window feedback and leaves fresh
  feedback; ledger byte-untouched (extend the existing discipline pins).
- `web/components/chat.test.tsx` + `feedback-buttons.test.tsx` (jsdom, action stubbed typed):
  buttons render only on answer messages with an auditId (⟨K:ui-tests-4⟩ incl. the
  `auditId: null` no-buttons pin AND the clarification-reply answer getting buttons); 👍 submits
  exactly once (double-click pin); 👎 opens the panel and submits with text; "Overslaan"
  submits bare; ok:false shows the soft copy AND resets busy (retry possible); verdict change
  re-calls the action and moves `aria-pressed`; refusal/clarification render NO buttons.
- ⟨K:migration-deploy-2⟩ **The pre-migration window pin**: helper-level — after
  `drop table answer_feedback` the upsert helper returns false (no throw); action-level — a
  throwing Db stub still yields `{ ok: false }`.

**AS-BUILT deviations (post-build diff review, 23 agents — 5 confirmed findings, all folded
same session):** (a) the upsert HELPER deliberately THROWS on a missing table (no `to_regclass`
guard — a plain query error); **the action's whole-body try/catch is the fail-soft guarantee**,

null/throwing auth, every attacker-input gate proven db-untouched — the pin this brief required
but the first build round skipped, plus a false coverage-claim comment corrected); the RETENTION
paths keep their own `to_regclass` guard (redaction must survive the window; a feedback click
may soft-fail). (b) The upsert-in-place pin lives in `tests/audit/feedback.test.ts` (incl. the
created_at-refresh assertion), not migration-017.test.ts. (c) The clarification-reply-path
button pin was also missing in round one — added (`chat.test.tsx`: clarification → reply →
answer → buttons wired to the anchor). (d) `dashboard.test.tsx`'s actions-mock needed the new
export (the recurring sibling-test-file lesson).
