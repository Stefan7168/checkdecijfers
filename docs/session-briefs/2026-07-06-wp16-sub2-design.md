## 0. The load-bearing design moves (read these first)

1. **The finder is an injected optional dependency of the answer pipeline.** `respond.ts` (and the policy layer under it) gains an optional `tableFinder` dep. Absent → today's behavior, byte-identical (unmatched term → the existing B15 clarification). Present (wired ONLY in `web/app/actions.ts`'s dependency construction) → the new flow. Consequence: the benchmark harness and every existing test construct the pipeline WITHOUT a finder and stay green untouched; B15 stays byte-identical BY CONSTRUCTION, not by luck. This is the single most important structural decision.
2. **The acknowledgment rides the refusal envelope** (new `RefusalReason` value `'onboarding_pending'`), exactly the ADR 022 meta-answer precedent. Consequence: `src/billing/gate.ts` is **byte-untouched** — its existing refusal branch auto-refunds the turn's 20-credit debit in full, which is precisely the wanted ledger shape. Do not modify gate.ts.
3. **The 100-credit debit + pending-row creation happen atomically in ONE transaction, in the ingestion module, called by the web action** after the gate returns — never inside `src/answer` (billing must not leak into the answer module: gate.ts's "wraps from the OUTSIDE" boundary).
4. **Delivery re-runs the original question through the full normal pipeline** (parse → query → compose → audit) after ingest + vocabulary registration. If that re-run does not produce an ANSWER, the request is `unanswerable` → full 100 refund. This check is the real verification gate: auto-generated registry vocabulary can only ever produce an answer that flowed through every existing validator, or a refund.

## 2. Trigger flow (CORE-1)

**Where:** `buildUnmatchedClarification` (src/answer/intent/policy.ts) is the single builder for the unmatched exit; its call sites (parse.ts, followup.ts, clarify.ts) all flow through it. Extend the policy layer so that WHEN a finder dep is present, the unmatched exit first calls `findTable(db, term)`:
- `confident` → check `pending_table_requests` for an active row (user, tableId): exists → refusal reason `'onboarding_already_pending'` (copy below); else → refusal reason `'onboarding_pending'` with the acknowledgment copy and a structured envelope field `onboarding: { tableId, topicTerm, confidence }`.
- `disclose` or `none` or finder throws → today's `buildUnmatchedClarification` output, byte-identical (pin this).
The finder call needs `db` + an async boundary at this point in the policy layer — **VERIFY** the call sites are async-capable (parse/followup/clarify are async); thread the optional dep through their existing deps objects, defaulting to absent.

**Copy (deterministic templates, Dutch):**
- `onboarding_pending` body, VERBATIM (owner-approved): *"Dat onderwerp staat nog niet in onze database. We vragen de cijfers nu automatisch op bij het CBS en controleren ze — meestal een kwestie van minuten. Je krijgt een e-mail zodra je vraag beantwoord kan worden. Heb je ondertussen nog een andere vraag?"*
- `onboarding_already_pending`: *"Deze cijfers worden al voor je opgehaald bij het CBS. Je krijgt een e-mail zodra je vraag beantwoord kan worden."* (No new debit — the action only triggers on `onboarding_pending`.)

**Envelope/R8:** add both reasons to `RefusalReason` (src/answer/respond/types.ts:28), a `kindWording` entry in refusals.ts, and the optional `onboarding` field on the refusal envelope. Extend `reconstruct.ts`'s refusal branch so the new reasons + field round-trip (envelope-integrity check covers them; no numbers involved so the no-unbacked-numbers belt is unaffected — but VERIFY the refusal-text number-scan whitelist tolerates none of the copy containing digits: neither string contains a digit, keep it that way). Web presentation: both reasons join the meta/smalltalk reason-aware list (#84) — plain info rendering, NO "Dit kon ik niet beantwoorden" header, NO geen-gok badge.

**Money (web action orchestration, actions.ts):** after `chargeAndRun` returns, if `response.kind === 'refusal'` with reason `onboarding_pending`:
`triggerOnboarding(db, { userId, requestId, questionText, auditId, onboarding })` → in ONE `withTransaction`: `pg_advisory_xact_lock(hashtext(userId))` (the reserveDebit pattern), balance check vs `getActionClassPrice(db,'heavy')` (=100), insert `credit_transactions` row (reason `onboarding_cost`, delta −100, request_id) — new ledger sibling `reserveOnboardingDebit` in ledger.ts (do NOT parameterize the existing hot `reserveDebit`) — then insert the pending row referencing the debit id. Result kinds: `started` | `insufficient {balance}` | `duplicate` (unique-index hit → return the acknowledgment again, no second debit).
- `started` → show the acknowledgment; cost display: the turn nets 0 (20 debited − 20 compensated by the gate) + 100 onboarding = **100 total**, and the UI cost caption must say 100, not 20.
- `insufficient` → show the EXISTING insufficient-credits UI shape (`required: 100`). **Documented decision:** the audited acknowledgment exists but is not rendered in this rare path; nothing fabricated, the fetch never started, ledger nets 0 for the turn. Record in the handoff.

**Ledger end-states (pin these exact nets in tests):** happy: −20, +20, −100 → net −100. Verification failure later: … +100 compensation → net 0. Insufficient at trigger: −20, +20 → net 0.

## 3. The cron job (CORE-2)

Module: `src/ingestion/onboarding.ts` (+ `onboarding-notify.ts`); thin route `web/app/api/onboarding-cron/route.ts`; config: **VERIFY** how the deploy serves the app (ci.yml deploy job runs `vercel build`/deploy from the repo root with `web/` as the project — put `{"crons":[{"path":"/api/onboarding-cron","schedule":"*/2 * * * *"}]}` in the file Vercel actually reads for this project: `web/vercel.json` if the Vercel project root is `web/`, else root `vercel.json`; confirm against `.vercel/project.json` or the deploy logs' project settings, record which in the handoff).

Route: if `process.env.CRON_SECRET` unset → 503 (fail closed, log plainly). Else require `Authorization: Bearer <CRON_SECRET>` → 401 otherwise. On auth: call the job with a deadline budget (~240s), return its summary JSON.

Job algorithm (single invocation):
1. **Reclaim:** `status='running' and claimed_at < now() - interval '20 minutes'` → back to `pending`, `attempt_count+1`. Rows with `attempt_count >= 3` → terminal `failed` + refund + notify (never infinite retry).
2. **Claim ONE row:** `update … set status='running', claimed_at=now() where id = (select id … where status='pending' order by created_at limit 1 for update skip locked) returning *`. No row → exit quietly. One table per invocation; the 2-minute cadence drains queues.
3. **Piggyback:** if `cbs_tables` already has the table registered + synced → skip to step 6.
4. **Size + slice (see §4):** estimate; over the 150k cap → build a sliced seed; else full.
5. **Register + sync** through the EXISTING pipeline (`registerTables` + `syncTable` with a runtime-derived `TableSeed` — **VERIFY** `registry-seed.ts`'s `TableSeed` shape and construct one from catalog metadata; the existing validators run inside `syncTable`). `outcome:'failed'` or a `needs_review` quarantine → failure path (step 8).
6. **Vocabulary:** insert `canonical_measures` (+ `default_coordinates` for the national total where a region dimension exists) rows for the synced measures, derived from the ingested CBS measure metadata — **VERIFY** the exact registry tables/columns in migrations/002 + src/registry/. Keys namespaced (e.g. `onboarded:<tableId>:<measureCode>`), definition labels = CBS's own labels verbatim (R10 spirit). Quality containment: step 7 refuses to deliver anything the full pipeline can't validate; a bad vocab row yields a refund, never a wrong number.
7. **Delivery:** re-run `question_text` through the full normal audited pipeline (the respond-audited entry, `source_tag: 'onboarding_delivery'`, NOT through the gate — the 100 already covers it; live LLM parse cost here is normal per-question spend). ANSWER → `delivered`, store `delivery_audit_answer_id`, notify. Refusal/clarification → `unanswerable` → refund 100 (existing `compensate`, idempotent) → notify honestly.
8. **Any failure** (fetch throw, ingest fail, quarantine): terminal `failed`, `failure_summary` in plain language, refund 100, notify. The job never lets an exception escape without the row reaching a terminal state + refund — wrap per-row processing in try/catch that finalizes.

Notification (`onboarding-notify.ts`): Resend HTTP API (`https://api.resend.com/emails`), from `noreply@mail.checkdecijfers.nl`, key from `RESEND_API_KEY` — absent → log + skip (best-effort; the dashboard is the record). Recipient: **VERIFY** email lookup — `select email from auth.users where id = $1` via the service-role connection should work; if not, record the alternative. Injected sender fn so tests stub it (the Stripe-signature test pattern). Three Dutch templates: delivered / unanswerable-refunded / failed-refunded.

## 4. Slice estimation (CORE-2)

Add `CbsSource.fetchObservationCount(tableId): Promise<number | null>` on BOTH `odata-v4.ts` (real: `GET {base}/{tableId}/Observations/$count` — **Assumption:** v4 supports `$count` on Observations; verify live in the supervised step; on 404/error return null) and `fixture-source.ts` (from fixture). Estimate = the count when available, else the product of dimension cardinalities from the table metadata the adapter already fetches for sync (**VERIFY** what `odata-v4.ts` exposes; add a metadata call only if truly absent). Over 150,000 (config constant `ONBOARDING_MAX_CELLS`, ADR 026): build the seed with a slice — the topic-matched measure(s) only + the national region code + the last 10 years (**VERIFY** how existing seeds express slices in `registry-seed.ts`; mirror that shape); record the slice choice in `failure_summary`-style plain language on the row (a `slice_note` text column is acceptable if cleaner — implementer's call, it's not money/correctness).

## 5. What the user sees

## 8. Top risks and their guards

1. **Auto-generated registry vocabulary produces a wrong mapping** → guard: delivery-must-answer through the full validator chain or refund (§3.7); quarantine via `needs_review` untouched; a wrong TABLE is already guarded by the finder's 0.8 confident floor + this same delivery check. Never a number outside the validated path (R1 holds by construction).
2. **Double-charging** (retries, double cron fire, double submit) → guards: unique (user,request) onboarding debit; unique active row per (user,table); `compensate`'s related-transaction idempotency; FOR UPDATE SKIP LOCKED claim; all four pinned by tests.
3. **Benchmark/B15 drift** → guard: finder-as-optional-dep; the finder-absent byte-identical pin; benchmark on the gate every stage.
