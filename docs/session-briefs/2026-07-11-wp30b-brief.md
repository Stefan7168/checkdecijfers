## What WP30b is

Make **"add source N" an executable contract instead of prose** (ADR 030 D6 as amended by A5/A6):

1. A **conformance harness** — a reusable, hermetic checker that a source passes on its recorded
   fixtures before it may exist: the FIVE existing validators + the contract-test families + the
   A6 catalog-lifecycle family, driven by a per-source **fixture manifest**.
2. The **two registry-consultation wirings** ADR 030 explicitly assigns to WP30b:
   - **A6**: `src/catalog/recall.ts` consults the registry's `currentCatalogStatuses` per row
     instead of the literal `'Regulier'`.
   - **ADR 030 § WP30a as-built (item 4 note)**: `src/query/run.ts`'s provisional boolean rule
     (`status !== 'Definitief'`, line 273) and the `freshestDefinitief` SQL literal (line 77)
     become registry-driven (`definitiveStatuses`).
3. The **"how to add a source" guide** (`docs/how-to-add-a-source.md`), referenced from the
   RUNBOOK and 04-architecture — written so a weaker-model executor can follow it (D6's stated
   purpose), including the honest list of **known WP30c wiring points** it must NOT attempt.

**Byte-identity is again the hard claim:** every existing answer, refusal, chart, citation, CSV,
stored audit row, benchmark result and LLM request byte stays identical. CBS resolves to today's
exact behavior; the harness only ADDS checks. Zero LLM bytes (A4), zero DDL (migration 016 stays
an unapplied file; nothing new is DB-shaped), zero prompt/fixture re-records.

## Work list (file-level)

### 1. `src/sources/registry.ts` — two additive fields on `SourceInfo` + the D4 helpers (pure leaf stays pure)

- `definitiveStatuses: readonly string[]` — CBS: `['Definitief']`. Doc comment: the fail-safe
  direction is **unknown status → provisional** (`!definitiveStatuses.includes(status)`) — a
  status we can't vouch for is marked, never silently definitive (principle c). Update the A2
  comment that currently says "stays run.ts's rule … adapter-contract material in WP30b".
- `isProvisionalStatus(info: SourceInfo, status: string): boolean` — the ONE rule, exported here
  (pure, no imports; unit-testable with a fake SourceInfo).
- `sourceKeyForTableId(tableId: string): string` — the D4 rule as code: prefix before the FIRST
  `':'` when present, else `'cbs'`. And `resolveSourceForTable(tableId)` =
  `resolveSource(sourceKeyForTableId(tableId))`. These live in registry.ts (pure, zero imports —
  the leaf discipline holds).

### 2. `src/query/run.ts` — registry-driven provisional rule (byte-identical for CBS)

- Line 273: `provisional: status !== 'Definitief'` →
  `provisional: isProvisionalStatus(resolveSourceForTable(q.tableId), status)`.
- Line 77: `` and status = 'Definitief' `` → `` and status = any($6::text[]) `` with
  `resolveSourceForTable(q.tableId).definitiveStatuses` as the sixth param. ⟨B3⟩ Use the
  codebase's `::text[]` cast convention, and give the definitief query its OWN params array
  (`[...params, statuses]`) — the two sibling freshness queries must not share one mutated array.
- **Pins:** existing query/answer/benchmark suites unchanged (the byte-proof); NEW unit tests on
  `isProvisionalStatus` with a fake source (`definitiveStatuses: ['Final']` marks `'Definitief'`
  provisional — proves registry-driven-ness); a freshness test proving `freshestDefinitief`
  matches pre-change behavior on the fixture db.
- **R8 note (verified first-hand):** `provisional` is STORED per cell; reconstruct re-validates
  from the stored result object and never recomputes it — old rows are untouched by construction.
  State this in the PR.

### 3. `src/catalog/recall.ts` — the A6 wiring (fixture-load-bearing SQL: byte-identical output, pinned)

- New builder (pure, exported for tests): given a sources map and a **starting parameter index**
  ⟨B3⟩ (recall.ts already threads dynamic indexes $1..$n terms + $n+1 limit — the builder's params
  append AFTER those, returned as `{ sql, params }` so no index collision is possible), emit the
  per-row is-current predicate as a **simple CASE over the derived source key**:
  ```sql
  case (case when position(':' in table_id) > 0 then split_part(table_id, ':', 1) else 'cbs' end)
    when 'cbs' then coalesce(status, '') = any($k::text[])
    -- one WHEN per additional registered source, statuses as array params
    else false
  end
  ```
  Source keys ordered deterministically (sorted); an id whose prefix is unregistered →
  `else false` → historic class (fail-safe: never fabricated "current"). The SQL derivation MUST
  agree with `sourceKeyForTableId` — pinned by a PGlite test comparing the SQL expression against
  the TS helper over bare / prefixed / colon-in-native-id samples.
- recall.ts replaces both `'Regulier'` literals (`is_regulier` alias may rename to `is_current`)
  with the builder's SQL; ranking, quotas, ordering and the rest of the query byte-unchanged.
- **Pins:** ALL existing recall.test.ts tests pass unchanged AND `find-replay.test.ts` replays
  every fixture (the byte-proof for the live shortlist — a membership/order shift would break the
  request hashes); NEW builder tests: (a) fake two-source registry + PGlite-seeded catalog rows
  with prefixed ids partition per-source; (b) a fake source whose current set ≠ `['Regulier']` is
  honored; (c) unregistered prefix → not current.

### 6. `src/sources/conformance.ts` (new) — the harness core

Pure, hermetic, no vitest dependency: `runSourceConformance(adapter, manifest, info)` →
`{ ok, failures: [{ family, tableId?, summary }] }` with **plain-language summaries in the
validate.ts style** (the owner can read what a candidate source got wrong).

Manifest (per source, committed next to its fixtures as `tests/fixtures/<key>/conformance.json` —
plain JSON, no comments; shape validated at runtime by the harness with loud errors):

```json
{
  "sourceKey": "cbs",
  "tables": [
    { "tableId": "03759ned" },
    { "tableId": "82235NED" },
    { "tableId": "85615NED", "schemaOnly": true }
  ],
  "declaredPeriodStatuses": ["Definitief", "Voorlopig", "NaderVoorlopig"],
  "declaredValueAttributes": ["None", "Impossible", "Confidential", "NotAvailable"],
  "declaredCatalogStatuses": ["Regulier", "Gediscontinueerd", "Vervallen"],
  "declaredDatasetTypes": ["Numeric", "Mixed", "Text"]
}
```

⟨B2⟩ The committed CBS manifest lists ALL 14 fixture tables, the 5 schema-only ones flagged, **no
slices** (measured: unnecessary). An optional per-table `slice` (the `CbsSlice` shape, applied via
the adapter's own `fetchObservations(id, slice)`) exists as an escape hatch for future sources
whose captures include statusless out-of-slice history — exercised ONLY by the harness's
failure-path tests, never by the CBS manifest.

Families (D6 as amended by A5 + A6):

- **F0 — registry-entry coherence** ("attribution fields present", widened): key === manifest key;
  displayName/attributionLabel non-empty; license present; `deepLink` null or returns an absolute
  `https://` URL embedding the table id verbatim; `provisionalDisplay` keys ⊆ declared period
  statuses; `definitiveStatuses` non-empty, ⊆ declared, **disjoint from provisionalDisplay keys**;
  `nullReasonLabels` keys ⊆ declared valueAttributes; `currentCatalogStatuses` non-empty ⊆
  declared catalog statuses.
- **F1 — fixture replay through the REAL parse code + D4 id discipline**: per table —
  `fetchTableSchema` (non-empty title, ≥1 measure, **exactly one TimeDimension** — the A7
  contract the fit gate's pre-checks key on); `fetchCodeList` per dimension non-empty;
  `fetchObservations` yields ≥1 row (unless `schemaOnly`); `fetchObservationCount` returns
  number|null without throwing; `fetchCatalog` parses with ≥1 entry (a catalog fixture is
  REQUIRED — the finder is how on-demand onboarding reaches a source). ⟨B4⟩ **D4 prefix contract:
  `sourceKeyForTableId(tableId) === manifest.sourceKey` for EVERY manifest tableId and EVERY
  `fetchCatalog` entry id** — one check proves both directions (cbs ids carry no `':'`; a
  non-cbs source's ids all carry its `'<key>:'` prefix; adapters speak prefixed ids NATIVELY —
  the waist never strips).
- **F2 — period-mapping round-trip**: every period code (TimeDimension code list + every
  observation period coordinate): `parsePeriodCode` ≠ null AND `encodePeriodCode(parsed)` ===
  the original code; every NON-NULL code-list status ∈ declared period statuses.
- **F3 — status/null-reason completeness**: every observation valueAttribute ∈ declared; every
  null-value row's attribute ∈ `nullReasonLabels` (R11: a refusal must be able to state the TRUE
  reason).
- **F4 — catalog-lifecycle completeness (A6)**: every catalog entry status ∈ declared ∪ {null};
  every datasetType ∈ declared ∪ {null}.
- **F5 — the five validators green** (registration semantics, per non-schemaOnly table): derive
  `RegistryUnits` via the same `unitsFromMeasures` pipeline uses (EXPORT it from pipeline.ts
  rather than duplicating), storedLabels from the fetched code lists, expected dimensions from
  the fetched schema; run checkSchemaFingerprint, checkRowPlausibility (lastRowCount null,
  default tolerance), checkPeriodParsing (the table's TimeDimension), checkDimensionMapping
  (acceptNewCodes false), checkUnitConsistency. For schemaOnly tables run the two row-free
  checks only. ⟨B5⟩ **Honesty note (in the harness doc comment AND the guide):** with
  self-derived expectations, the fingerprint and unit checks prove "the pipeline's gates ACCEPT
  this adapter's shapes" (registration semantics — same trust-on-first-use as registerTables),
  NOT drift detection; drift detection remains sync-time work against a stored registry. F5's
  real bite here is period parseability, publication+status coverage, no reason-less nulls, no
  duplicate cells, coordinate↔code-list closure.

⟨B6⟩ **Known WP30c wiring points** (a dedicated guide section — real, verified landmines a
source-N executor must know are NOT yet wired; none are WP30b work):

1. `ingestCatalog`'s prune is not source-scoped (src/catalog/ingest.ts:66) — a second source's
   refresh would WIPE the other's mirror rows; WP30c must scope the prune (id-prefix or the
   migration-016 source column once applied).
2. `recall.ts` filters `language = 'nl'` — a non-Dutch-language source's catalog rows are
   invisible to the finder (interacts with the #131 i18n lane).
3. `template.ts:28` / `refusals.ts` resolve display wording via `resolveSource(undefined)` —
   correct while single-source; WP30c threads the result's actual source into compose display.
4. The onboarding-cron route constructs its adapter directly (the ⟨B1⟩ deferral) — WP30c wires it
   through `adapterFor` by the table's source.
5. No conformance family checks GeoDimension codes against the Dutch region taxonomy (D2) — fine
   for a national-only source; a REGIONAL source needs that check designed with its adapter.
6. `datasetType <> 'Text'` and the region-prefix tables stay canonical-waist vocabulary the
   adapter maps INTO.

Plus: **what you must NOT touch** (compose/validate/billing/prompts — the waist's whole point),
and the fail-direction table (display falls back to cbs / ranking treats unknown as not-current /
