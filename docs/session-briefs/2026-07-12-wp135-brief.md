## 1. Migration `migrations/019_chat_threads.sql` (hermetic file only — no live DDL)

```sql
-- chat_threads stores WHEN, never WHAT: no text columns, by design (ADR 033 D1).
-- No GRANT/RLS statements needed: migration 003's defaults lock this table (011/018 posture).
create table chat_threads (
  id bigint generated always as identity primary key,
  user_id uuid not null,
  created_at timestamptz not null default now(),
  last_activity_at timestamptz not null default now()
);
-- guarded FK to auth.users, conditional on auth schema existence (migration 005 pattern)
-- ⟨A2⟩ the guarded FK is REAL DDL, not a comment (migration 005's exact pattern, retargeted):
do $$
begin
  if exists (select 1 from information_schema.tables
             where table_schema = 'auth' and table_name = 'users') then
    execute 'alter table chat_threads add constraint chat_threads_user_id_fkey
             foreign key (user_id) references auth.users(id)';
  end if;
end $$;

create index chat_threads_by_user_activity on chat_threads (user_id, last_activity_at desc);

alter table audit_answers add column thread_id bigint references chat_threads(id);
-- no ON DELETE clause: redact-not-delete posture (migration 005 precedent)
create index audit_answers_by_thread on audit_answers (thread_id) where thread_id is not null;
```

⟨A2⟩ **PR-review checklist line (CI is blind here):** the hermetic PGlite harness has no `auth`
schema, so the guard always no-ops in tests — a missing FK block is invisible to CI. The PR
description must show a human-readable diff of migration 019's DO-block against migration 005's,
and the owner-supervised go-live step verifies the constraint exists on prod after applying.

## 2. Backend: `web/backend/threads/index.ts` (new, mirrors `billing/history.ts` conventions)

All SQL with **bound parameters**; every query scoped `user_id = $n`. Note: `web/backend` is the
`../src` symlink, so this module physically lives at `src/threads/` — a NEW top-level module;
record the addition in ADR 001's as-built module list in the same change. Functions:

- ⟨A1⟩ `validateThreadOwnership(db, userId, rawThreadId)` → validated thread id **or null** —
  **read-only, never an INSERT**. Null, malformed, or **not owned by userId** → null (fail-safe,
  the ADR-021 invalid-context treatment; a forged id NEVER attaches rows to another user's
  thread and never errors in a way that leaks existence).
- ⟨A1⟩ `attachOrCreateThread(db, userId, validatedThreadIdOrNull, auditId)` → ONE atomic step,
  called ONLY on a gated-ok outcome with an audit id: create the `chat_threads` row if the id is
  null, then `UPDATE audit_answers SET thread_id = $1 WHERE id = $2 AND user_id = $3 AND
  thread_id IS NULL` + touch `last_activity_at`, in the same transaction. **Thread creation is
  thereby lazy BY CONSTRUCTION** (ADR 033 D1: no empty threads): insufficient_credits,
  duplicate_request, the ⟨W4⟩ early return, and thrown pipeline exceptions never reach this call
  and thus never create a row.
  **Deliberate design: the answer pipeline (`src/answer/**`) is NOT touched** — thread attach is
  a post-hoc UPDATE after the pipeline returns (UPDATE on audit rows is precedented by
  redaction). If the attach fails, the answer stands threadless (degraded, logged) — never block
  or roll back an already-audited answer.
- `listThreads(db, userId, limit=50)` → `{id, title, lastActivityAt}[]`. Title = first
  **non-redacted** row's `question` (`question <> REDACTED_QUESTION_TEXT`, order created_at asc,
  id asc), truncated to 60 chars in TS. Threads whose every row is redacted (title source NULL)
  are **filtered out** — self-service deletion and the 2-year purge empty the sidebar with zero
  new machinery (ADR 033 D2). Import the sentinel from `src/answer/audit/retention.ts` — never
  duplicate the string.
- `getThreadRows(db, userId, threadId)` → the thread's audit rows (created_at asc, id asc):
  `id, kind, question, final_text, reply_text, created_at, response` (full envelope) — same
  zero-loss posture as `history.ts` — **⟨A3⟩ plus `creditsCharged` per row via the same
  debit/compensation ledger join `history.ts` already implements** (reuse/share that join; the
  live `message.cost` caption comes from `gated.netCost`, which is NOT in the stored envelope,
  so replay must recompute it from the ledger or the caption silently vanishes on resume).

## 3. Replay + context rebuild (deterministic, zero LLM) — layering per ⟨A3⟩

**⟨A3⟩ Two-layer split (the `web/backend → ../src` symlink makes this mandatory: src code cannot
import `web/lib/*`):**

- `web/backend/threads/replay.ts` (= `src/threads/replay.ts`) produces **structural replay
  parts** per row: message texts, `answerView` parts from `response.answer` (only when the
  structural fields are present — the #115 zero-loss rule, else the `final_text` blob), `chart`,
  `suggestions`, `webSection`, `provisional`, `auditId = row.id`, the raw envelope, and
  `creditsCharged` (from §2's join). **R8 pin: replayed message text is byte-equal to stored
  `final_text`.**
- The **full `ChatMessage` assembly lives web-side** (`web/app/actions.ts` + a small
  `web/lib/replay-assemble.ts`): it calls the SAME live-path functions —
  `buildCitation` (web/lib/citation.ts), `buildAnswerCsv` (web/lib/csv.ts), `statCardData`
  (web/lib/stat-card-data.ts) — over each replayed envelope, so `citation`/`csv`/`card` (and
  thereby the **dock-tab eligibility** of stat-card answers) reconstruct identically to live.
  The `kind` reclassification (refusal reasons meta/smalltalk/onboarding_* → 'info') is
  **EXTRACTED from chat.tsx's inline receive-handler into one shared web/lib helper** used by
  BOTH the live path and replay — byte-identity by construction, not by parallel copies.

**Turn rules within a row (⟨A4⟩):** each row emits **exactly ONE user-turn message** — from
`reply_text` when it is non-null (a reply row's `question` column echoes the ORIGINAL question,
per the respond-audited.ts convention history.ts documents), else from `question` — followed by
its assistant message. A resumed clarification round therefore replays as exactly
[question → clarification → reply → outcome], never with the original question duplicated.

**Redacted rows (⟨A7⟩):** detect via `row.question === REDACTED_QUESTION_TEXT` (import the
sentinel, never duplicate the string) and emit ONE placeholder message ("Deze vraag is
verwijderd." styling, no user+assistant sentinel pair — the chat-side equivalent of the
dashboard's isDeleted posture, which is a new small branch, not an existing component).

- `rebuildContext(rows)` → run the existing `buildConversationContext` (src/answer/context/
  build.ts) over the LAST envelope that yields non-null, walking backwards, **⟨A7⟩ skipping
  redacted rows BEFORE the call** (a redacted 'answer' envelope keeps `kind='answer'` but has no
  `result`, and `resolvedIntent` would throw rather than return null — the same guard
  verify-audit-rows.ts already applies); then `validateConversationContext` against the live
  registry. Any parse/validation failure → null (standalone parse — honest degradation,
  ADR 021). No new context logic is written; only the existing deterministic functions are
  called.

## 4. Server actions (`web/app/actions.ts`) — all additive, `?? null`-guarded (WP129 pattern)

- `askQuestion(question, requestId, rawContext?, rawSelection?, rawThreadId?)`: 5th optional
  param. Flow ⟨A1⟩: `currentUserId()` → `validateThreadOwnership` (read-only) → existing
  pipeline call UNCHANGED → **only on a gated-ok outcome with an audit id**,
  `attachOrCreateThread`. Return gains `threadId` (null when nothing attached). Absent param =
  today's behavior byte-identical (benchmark/runner paths never send it; **benchmark rows keep
  `thread_id` NULL** — pinned).
- `replyToClarification(pending, reply, requestId, rawSelection?, rawThreadId?)`: same ⟨A1⟩
  treatment. **⟨A6⟩ The reply's `rawThreadId` is the threadId the CLIENT captured alongside
  `pending` at question time** (stored together in chat state), NOT the sidebar's current
  `activeThreadId` — so a reply always attaches to its originating thread; on
  ownership-validation failure it attaches to a fresh thread, never cross-attaches.
- New `listMyThreads()` and `loadMyThread(rawThreadId)` — `getClaims()`-verified like
  `deleteMyQuestionHistory`; `loadMyThread` returns `{threadId, messages, context}` and returns
  an empty result (never an error leaking existence) for a thread not owned by the caller.
- New `signOut()` — wraps Supabase sign-out, redirects to `/login` (WP24 scope, ADR 033 D6).

## 5. Web UI (flag-gated at the page level; `web/app/layout.tsx` untouched)

`web/app/page.tsx`: `WORKSPACE_ENABLED !== '1'` → render today's `<Dashboard>` **byte-identically**
(no new props, no thread calls — the WP129 dormancy discipline). **⟨A5⟩ The flag gates EVERY
WP135 surface, not just `/`:** flag off → `/credits` renders exactly as today (no header),
`/login` renders exactly as today (no stripped header), and `/geschiedenis` redirects to `/`
(the route ships dark). The `signOut` server action may exist in code but NO UI reaches it when
the flag is off. Flag on → `<Workspace>`:

- **`web/components/workspace.tsx`** — holds `balance` (the #68 live-update pattern moves up from
  Dashboard), `activeThreadId`, thread list state; grid: sidebar (collapsible) | chat column |
  visual dock (only ≥ `lg` AND ≥1 visual). Renders `SiteHeader`, the purchase-success banner
  (relocated from Dashboard), and the footer line (ADR 033 D6 copy: `Cijfers: CBS StatLine
  (CC BY 4.0) · Elk getal herleidbaar tot een officiële CBS-tabel · Over dit project` — NO
  privacy link until #14(d) exists; "Over dit project" links to a static page/anchor with
  01-product-vision language).
- **`web/components/site-header.tsx`** — wordmark → `/`, live balance chip (ADR 006: read live,
  never hardcoded), "Credits kopen" → `/credits`, "Geschiedenis" → `/geschiedenis`, account menu
  (signOut + relocated `DeleteHistoryButton`). Stripped variant (wordmark only) on `/login`.
- **`web/components/thread-sidebar.tsx`** — "Nieuwe chat" button on top; threads grouped
  client-side: vandaag / gisteren / afgelopen 7 dagen / ouder (by `lastActivityAt`, Europe/
  Amsterdam day boundaries). Click → `loadMyThread` → chat receives `{messages, context,
  threadId}` — **⟨A6⟩ and a sidebar thread-switch CLEARS `pending` (and its captured threadId),
  exactly like "nieuwe chat" does**: an abandoned clarification stays abandoned (same as a page
  reload today), it never follows the user into another thread. "Nieuwe chat" = the ADR-021
  explicit reset: messages `[]`, pending null, **context null**, input cleared, `activeThreadId`
  null (thread created lazily on the next completed question); `selectedSources`/`webSelected`
  survive.
- **`web/components/visual-dock.tsx`** — tabs derived FROM the messages array (every message
  with `chart` or a stat-card-eligible answer ⇒ one tab, newest active; labels deterministic:
  `Grafiek n` / `Kaart n` + truncated question — no LLM). Tab state is never persisted; replay
  reconstructs it. Renders the SAME `ChartView`/`StatCard` components, internally unchanged.
- **`chat.tsx`** — minimal diff: accepts optional `dockMode` + `onThread(threadId)` + initial
  `{messages, context, threadId}` props; when `dockMode` is active a visual renders as a small
  in-flow reference chip ("in het paneel", activates its tab) instead of inline — **each visual
  renders exactly once** (ADR 033 alt-4 rejection). Below `lg` (client media query) visuals stay
  inline exactly as today. **The message envelope block is byte-identical in every mode**:
  suggestion chips, feedback buttons, source-selection chips, attribution chip, provisional
  badge, staleness/definition/marking lines, refusal header, citation-copy + CSV buttons,
  pre-send cost line, and the web section (ALWAYS last-in-bubble, NEVER docked — ADR 032).
- **`web/app/geschiedenis/page.tsx`** — auth-guarded page hosting today's `QuestionHistory`
  (server component moved, not rewritten; the #74 "mijn aanvragen" seam lives here later).
  `/credits` gains the header like other authenticated pages.

## 6. Test pins (the gate; add to existing suites, hermetic PGlite)

1. **Cross-user isolation:** `listThreads`/`getThreadRows`/`loadMyThread` return nothing for
   another user's thread; forged `rawThreadId` on ask/reply NEVER attaches a row to another
   user's thread (fresh thread instead). Mirrors the #14 cross-user pins.
2. **R8 replay:** replayed text byte-equal to stored `final_text`; answerView parts only when
   structural fields present (zero-loss rule).
3. **Context rebuild determinism:** equals `buildConversationContext` over the last eligible
   envelope; unparseable/stale envelope → null; validation failure → null. **⟨A7⟩ A redacted
   'answer' row inside an active thread does NOT crash the rebuild — it is skipped.**
4. **Redaction:** fully-redacted thread vanishes from `listThreads`; partially redacted thread
   titles from the first non-redacted row; existing retention suite untouched and green.
   **⟨A7⟩ A redacted row replays as ONE placeholder message, never a user+assistant sentinel
   pair.**
5. **Lazy creation ⟨A1⟩:** no thread row until a first completed question; "nieuwe chat" alone
   creates nothing; **insufficient_credits, duplicate_request, the ⟨W4⟩ early return, and a
   thrown pipeline exception on a brand-new chat leave the `chat_threads` row count unchanged**;
   `last_activity_at` touched on ask AND clarification reply.
6. **Dormancy ⟨A5⟩:** flag off → `/` Dashboard render byte-identical AND `/credits` + `/login`
   byte-identical to today AND `/geschiedenis` redirects to `/`; no thread action and no shell
   surface reachable anywhere.
7. **Envelope byte-identity:** rendering test that the message envelope block (incl. web section
   last-in-bubble) is identical in dock mode vs inline mode vs today.
8. **Benchmark neutrality:** runner path sends no threadId; benchmark rows `thread_id` NULL.
9. **Shell (flag on):** signOut action; header presence rules (stripped on `/login`, present on
   `/`, `/credits`, `/geschiedenis`); footer static content EXACT string (byte-pinned; owner
   gives the final look at PR review).
10. **Migration 019** runs green in the hermetic harness (auto-included).
11. **⟨A4⟩ Clarification-round replay:** a thread holding one clarification round replays as
    exactly [question → clarification → reply → outcome] — message count pinned, the original
    question never duplicated.
12. **⟨A3⟩ Replay completeness:** a replayed stat-card-eligible answer reconstructs `card`,
    `citation`, `csv`, and `creditsCharged` identical to its live render, and produces the same
    dock tab; a replayed meta/smalltalk refusal reclassifies to 'info' via the shared helper.
13. **⟨A6⟩ Reply-thread binding:** a reply sent after the sidebar switched threads attaches to
    the ORIGINATING thread of its clarification, never the newly active one (and pending is
    cleared on switch, so the UI cannot even offer it).
