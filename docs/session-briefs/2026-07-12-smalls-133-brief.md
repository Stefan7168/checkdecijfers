## 1. #133 — reconstruction policy (historical rules + redacted rows)

**Policy decision (a) — historical-behavior versioning: NO versioning machinery.** Reconstruction verifies under TODAY's deterministic rules. When a builder's behavior legitimately changes (a safety fix like #64's non-contiguous-chart refusal or #115-lever-a's circular-title suppression), rows stored under the older rule may diverge on re-derivation. R8's core promise is untouched — the stored `final_text`/`response` IS what the user saw, verbatim. Divergence against a later, *safer* rule is an accepted, **documented and pinned** gap, not a bug. Rationale: honoring a row's own historical rule would mean keeping every superseded builder behavior alive forever (a growing maintenance trap) for zero user value; all schema versions remain 1 and `reconstructionReport` stays reject-on-mismatch (ADR 016 as-built note added).

**Mechanism: a known-divergence register, pinned not blanket-skipped.** New committed file `scripts/audit-known-divergences.json`: entries `{ id, kind, expectProblemsContaining: string[], cause, recordedDate }` — today exactly two entries: row 76 (`chart spec does not re-derive` — #64, session 22) and row 227 (`definition line does not re-derive` — #115-lever-a). `scripts/verify-audit-rows.ts` treats a listed row as clean ONLY IF every reported problem matches one of the entry's expected substrings; any unexpected problem still fails (exit 1). Unlisted divergences fail as today. A register entry that no longer matches (row now reconstructs clean, e.g. after redaction) is reported as stale, not an error.

**Policy decision (b) — redacted rows get a real check, not a skip.** New exported `redactionIntegrityReport(record)` in `src/answer/audit/retention.ts` (the module that owns the sentinel shape — single source of truth): verifies the row matches the EXACT expected redacted shape — `question`/`finalText` === `REDACTED_QUESTION_TEXT`; `response` deep-equals `redactedResponse(kind)` for the row's kind (no extra keys — a leftover `answer`/`result` key is a failed redaction); `intent`/`intentHash`/`conversationContext` null; `resultIds`/`tableIds` empty; `tables` empty; `replyText` null-or-sentinel and `pendingClarification` null-or-`redactedPendingClarification()` shape, paired. Returns `{ok, problems}` like `reconstructionReport`. `verify-audit-rows.ts` routes `redacted: true` rows here instead of skipping: summary becomes "N/M reconstruct clean + K/K redacted rows redaction-verified". `reconstructionReport` and the benchmark gate stay byte-untouched (benchmark rows are never redacted — purge scopes exclude `benchmark`/`validation`).

**Tests:** tamper tests proving each redaction-integrity check can fail (house style, mirrors `audit-records.test.ts`); an end-to-end PGlite test: store → redact via `deleteUserQuestionHistory` → `redactionIntegrityReport` ok; register-mechanism unit test (matching problems pass, unexpected problem fails, stale entry flagged).

**Docs:** ADR 016 as-built note (the policy paragraph), open-questions #133 → resolved, 05-data-rules R8 row note.

**Invariants at stake:** R8 (reconstructionReport untouched — diff-proven), benchmark gate unchanged, no schema/DDL.

## 2. #119 — delivery idempotency (deliver→finalize crash window)

**Fix:** recovery check in BOTH places a `running`/stale row re-enters processing, using the recorded sketch (existing `onboarding_delivery` audit row for this `request_id`):

- New store helper `findDeliveryAuditAnswerId(db, requestId): Promise<number | null>` — `select id from audit_answers where source_tag = 'onboarding_delivery' and request_id = $1 order by id asc limit 1` (order pinned: historical double-rows resolve deterministically to the FIRST delivery).
- `processOneRow` (src/ingestion/onboarding.ts:275): at the top, before any ingest/fit work — if found: `finalizeDelivered(db, row.id, { deliveryAuditAnswerId: existing })`, send the `delivered` notify (the crash happened BEFORE the first notify, so this is exactly-once from the user's view), return `'delivered'`. No pipeline re-run, no second audit row, no ledger touch.
- `runOnboardingJob`'s exhausted loop (onboarding.ts:445–467): before `failAndRefund` — same check; if found, finalize-delivered instead of refunding. Without this, a crash row that exhausts `MAX_ATTEMPTS` gets refunded despite the user having received the answer = the user keeps answer + refund (a real, if tiny, money leak the sketch missed).

**Explicitly NOT touched:** `src/billing/` primitives, `chargeAndRun`, the refund paths' transactions, the claim SQL/`SKIP LOCKED`, `STALE_RUNNING_MS`/`MAX_ATTEMPTS`. The deliver→finalize pair itself is NOT wrapped in a transaction (the pipeline call must not run inside a tx; the idempotency check makes the crash window safe instead — same shape as `chargeAndRun`'s documented posture, gate.ts:35).

**Tests (PGlite, existing harness):** (1) crash simulation — deliver via the real pipeline, revert row to `running` with stale `claimed_at` (simulating crash before finalize), re-run job: asserts exactly ONE `onboarding_delivery` audit row for the request_id, row finalized `delivered` pointing at the EXISTING audit id, exactly one notify, ledger byte-untouched; (2) exhausted-path recovery — same setup with `attempt_count = MAX_ATTEMPTS`: finalized delivered, NO compensation row, no `failed` status; (3) no-false-positive pin — a fresh pending row with no prior audit row delivers normally (the check must not fire on first attempts).

**Invariants at stake:** R8 (audit-before-shown ordering unchanged on genuine first attempts), the "terminal + refunded is atomic" invariant (untouched), ledger-untouched.

## 3. #120 — GDPR retention: onboarding data legs

**Fix, both paths (`deleteUserQuestionHistory` + `purgeExpiredQuestionHistory`, src/answer/audit/retention.ts:166–190):**

1. **Widen the audit redaction scope** from `source_tag = 'user'` to `source_tag in ('user', 'onboarding_delivery')` (both WHERE clauses; the purge's `answer_feedback` subselect widens identically). `benchmark`/`validation` stay excluded — the #14 pins extend to prove it.
2. **New `pending_table_requests` leg inside the SAME transaction** (extending the `redactMatchingRows` extra-statement mechanism that WP128's feedback-delete introduced): `update pending_table_requests set question_text = $s, topic_term = $s, failure_summary = case when failure_summary is null then null else $s end where user_id = $1` (self-service) / `where created_at < $cutoff` (purge), `$s` = `REDACTED_QUESTION_TEXT`. All statuses redacted, including `pending`/`running` (below). `slice_note`, ids, money columns untouched — the financial/status skeleton stays, mirroring the audit-row posture.
3. **`scripts/gdpr-purge.ts` dry-run** widens to match: audit count over both source_tags + a `pending_table_requests` count — the preview must never under-report what `--apply` would touch.

**Accepted, documented limitation (in-flight race):** a redaction racing a `running` onboarding row doesn't stop the job — its in-memory `questionText` copy still delivers, and the delivery audit row written AFTER the redaction survives until the next self-service click or purge (which now cover `onboarding_delivery`). A redacted `pending` row later claimed runs the sentinel text through the pipeline → refuses → refunds (fail-safe). Recorded in open-questions as a #120 residual; exposure today nil (owner is the only user).

**Dashboard note:** the non-delivered onboarding branch (`question-history.tsx:138–164`) renders the sentinel text verbatim — acceptable copy for a user who deleted their own history; no component change in v1 (kept out of scope; the branch gets a residual note).

**Tests (mirror the #14 pins, same file conventions):** cross-user pin for the pending leg (other user's `pending_table_requests` rows byte-untouched); `onboarding_delivery` audit rows redacted by BOTH paths while same-user `benchmark`/`validation` rows survive; purge cutoff strictness on the pending leg; idempotency; ledger byte-untouched with the new legs in play; the WP128 feedback interplay still green.

**Docs:** 05-data-rules retention paragraph (scope now includes onboarding), 04-architecture "question text in exactly one place" seam note (now two write points, ONE retention seam covering both), open-questions #120 → built + residual.

**Invariants at stake:** #14 cross-user scoping, ledger-untouched, `reply_round_complete` pairing (reused mechanism), R8 (redacted onboarding rows must pass the NEW `redactionIntegrityReport` from change 1 — cross-test).

**Tests:** link present in all three outcome bodies with origin set; absent when origin null; trailing-slash normalization; existing template pins unchanged.

**Invariants:** none touched (display-only, best-effort email path); dashboard deep-link-per-answer stays future work (#116 recorded the dashboard link as the cheap first step).
