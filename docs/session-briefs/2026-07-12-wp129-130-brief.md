## What this WP is

Two owner requests built as ONE work package (owner-confirmed, Q10):

1. **#129 — source-tags chips in the chatbox.** A "CBS data" chip (pre-checked, rendered FROM the
   source registry) plus a new **"Internet"** chip (the augmentation channel, NOT a registry
   source). The selection is a **structural input** to the pipeline (never prompt text), rides the
   audit envelope so R8 reconstructs what was searched, and deselecting everything yields an
   honest refusal — never a guess.
2. **#130 — web search as a separated, unverified augmentation channel.** When the Internet chip
   is on, a **new, self-contained LLM call** (Anthropic native `web_search_20260209` server tool)
   produces ≤4 one-sentence findings with citations, rendered in a visually distinct block —
   header **"Van het web (niet door checkdecijfers geverifieerd)"** — BELOW the validated CBS
   body (or below the refusal text). Web findings NEVER enter the validated body, any validator
   input, `attribution`, chart data, conversation context, or the benchmark's fabrication scoring.
   **The separation IS the honesty model.**

Money: **+10 credits** when the Internet chip is on, as a **separate ledger debit** with its own
reason (`websearch_cost`, the WP16 `onboarding_cost` pattern), refunded automatically via the
EXISTING `compensate()` primitive whenever no web section is delivered. `src/billing/gate.ts` and
every existing ledger function stay **byte-untouched**.

## The invariants at stake (name them in every sub-brief and review)

- **R1/R3 + benchmark fabrication scoring:** the web section's text must NEVER appear in
  `response.text` / `answer.body` — it rides a separate structural envelope field. The 20-task
  benchmark runs with web search structurally OFF and must stay green UNCHANGED (0 fabricated).
- **R2:** the web call is a NEW prompt in a NEW module (`src/websearch/`) — the ONE place new
  prompt bytes are allowed in this WP. The existing intent/clarify/followup/compose/rerank/fit
  prompts stay byte-identical; every existing LLM fixture replays without re-record (the proof).
- **R8 + A1 discipline:** the web section and the chip-selection state are stored VERBATIM as
  additive envelope fields with `?? null` reads + absent-key regression tests (four prior R8 bugs
  in this class — #133). Reconstruction of a web section = replay of stored bytes, never
  re-derivation (the web is non-deterministic).
- **Money invariants:** debit-before-spend; one idempotent debit per (user, requestId) per
  reason; one compensation per debit (`related_transaction_id` key); the append-only ledger; the
  gate's netCost always mirrors the compensation actually applied. Billing gate/ledger primitives
  byte-untouched — the add-on is a SIBLING debit (additive functions), exactly like WP16's.
- **Principle (c):** deselecting all sources → deterministic refusal; web failure → the CBS
  answer always ships + one honest line + automatic refund. Fail-soft everywhere on the web path;
  fail-closed audit semantics unchanged.

## Measured facts the design rests on (scouted + session-verified — do not re-derive)

- **SDK:** `@anthropic-ai/sdk` ^0.110.0 is installed in BOTH package.json files and its installed
  `messages.d.ts` fully types the tool: `WebSearchTool20260209` (`{ name: 'web_search', type:
  'web_search_20260209', max_uses?, user_location?, allowed_domains?/blocked_domains? }`),
  `UserLocation` (`{ type: 'approximate', city?, country?, ... }` — country = two-letter ISO),
  `WebSearchToolResultBlock` (`content: WebSearchToolResultError | WebSearchResultBlock[]` — an
  in-band error is an OBJECT, a success is an ARRAY; branch on that before indexing),
  `WebSearchToolResultErrorCode` (`invalid_tool_input | unavailable | max_uses_exceeded |
  too_many_requests | query_too_long | request_too_large`), and `CitationsWebSearchResultLocation`
  (`{ cited_text, title, url, encrypted_index, type: 'web_search_result_location' }`) on
  `TextBlock.citations`. **No SDK upgrade needed.**
- **Server-tool semantics:** errors come back in-band (HTTP 200); search-result content bills as
  input tokens; citations are always enabled and token-free; a server-tool loop can end with
  `stop_reason: 'pause_turn'`; models can also return `stop_reason: 'refusal'`. The
  `web_search_20260209` (dynamic filtering) variant requires a mid-tier model — cheap-tier models
  only support the basic 20250305 variant (this is why ADR 032 fixed mid-tier).
- **LLM client:** `AnthropicLlmClient` (src/answer/llm/client.ts) does NOT pass `tools` and its
  `LlmRequest` hash keys every recorded fixture — **do not touch it**; the web channel gets its
  OWN client + interface in `src/websearch/` (hermetic via a hand-authored fake, not the
  fixture-replay machinery).
- **Billing gate (src/billing/gate.ts):** pre-flight debit of the 'simple' class (20 credits) via
  `reserveDebit` (per-user advisory lock, idempotent on `(user_id, request_id) where
  reason='question_cost'`); compensation AFTER the pipeline based on the RETURNED
  `response.kind` — clarification → refund down to 10, ANY refusal → full refund, exception →
  full refund + rethrow; `netCost` mirrors the compensation applied.
- **Ledger DDL:** `credit_transactions_reason_check` currently allows `('signup_grant',

  `credit_transactions_validate_compensation` trigger only allows compensations reversing
  `question_cost`/`onboarding_cost` (widened by 013 — the exact precedent our migration follows);

## Owner decisions → mechanics (worked-out consequences, stated honestly)

| Decision (ADR 032) | Mechanical consequence |
|---|---|

| Web-only mode allowed (Q4) | CBS deselected ⇒ deterministic pre-parse refusal (new reason `web_only`) + web section below it. The gate's EXISTING refusal-refund makes the turn net exactly 10 (the add-on) — zero LLM calls, zero gate changes. **Worked-out consequence, flag in the PR: web-only NETS 10, not 30** (nothing CBS-shaped ran; charging the base for it would be charging for nothing) — ⟨W4⟩ but a balance of 30 must be AVAILABLE transiently (the untouched gate's pre-flight hold). |
| Refusal + web coexist (Q5) | The web call runs on refusal outcomes too (skip-list below); refusal text byte-untouched; section renders below it. Net: 0 (base refunded) + 10 (web kept) = 10. |
| Fail-soft + auto refund (Q6) | **One rule: the add-on is kept iff the returned envelope carries a web section with ≥1 cited finding; EVERY other outcome (API error, in-band error object, `refusal`/`pause_turn`/other stop, zero cited findings, timeout, insufficient balance at reserve time, audit fail-closed substitution) ⇒ automatic `compensate()` of the web debit + one honest line.** The CBS answer always ships regardless. |
| Clarification turns | The web call is SKIPPED on `kind === 'clarification'` (searching an ambiguous question is spend without value); the lazy reserve never fires ⇒ nothing charged, nothing to refund. | No web on smalltalk/meta/internal/onboarding | ⟨W3⟩ Refusal-reason skip-list `{'smalltalk','meta','compound','internal','no_sources','onboarding_pending','onboarding_already_pending'}`: not data questions, internal errors, or fetch acknowledgments — web skipped, reserve never fires, no charge. The onboarding pair was ADDED by the review: the acknowledgment ("je tabel wordt voorbereid") is not a data refusal, renders as ChatMessage kind 'info', and including it created a 110-credit combined turn with netCost/settlement interplay against `maybeTriggerOnboarding`'s gated-replacing branches — all of which evaporate by skipping. **Flag to the owner in the PR** (Q5's "may still render" gives this latitude). | ⟨W4⟩ Minimum balance for ANY web-opted turn = 30 | The untouched gate ALWAYS holds the base 20 before the pipeline runs, and the +10 web reserve happens INSIDE the pipeline (before the gate's refund posts) — so a web-opted turn transiently needs balance ≥ 30 in BOTH modes, even though web-only NETS 10. Owner-facing copy states the net cost per mode AND the ≥30 requirement honestly. **Flag to the owner in the PR** (consequence of the gate-untouched constraint, not a choice). |

| max_uses 3, NL, open web (Q7/Q9) | Tool param verbatim: `{ type: 'web_search_20260209', name: 'web_search', max_uses: 3, user_location: { type: 'approximate', country: 'NL' } }`. No domain lists. |
| Header + shape (Q1/Q3) | Header string constant `"Van het web (niet door checkdecijfers geverifieerd)"`; ≤4 findings, one sentence each, domain-name-only links; bordered muted block; never above the CBS body; no numbers in bold, no charts. |
| Sequencing (Q10) | Chips first (structural selection end-to-end with web:false), then the web channel, same branch/PR. |

### 3. `src/websearch/` — the NEW module (the one place new prompt bytes are allowed)

- `types.ts` (pure leaf): 
  ```ts
  export interface WebFinding { text: string; citations: { url: string; title: string | null }[] }
  export type WebSection =
    | { status: 'ok'; findings: WebFinding[]; model: string; searches: number;
        usage: { inputTokens: number; outputTokens: number }; promptVersion: number }
    | { status: 'failed';
        code: 'api_error' | 'no_findings' | 'stopped' | 'insufficient_balance' | 'not_configured' };
  ```
  ⟨W6⟩ `not_configured`: selection says `web: true` and the response kind/reason OWES a web
  attempt, but no client/billing was injected — attach records the skip instead of leaving the
  key null, so "owed but none recorded" is impossible BY CONSTRUCTION and reconstruct check (d)
  (work item 5) is exact, never heuristic.
  ```ts
  export interface SourceSelection { sources: string[]; web: boolean }
  ```
- `client.ts`: `export const WEBSEARCH_MODEL = 'claude-sonnet-5';` (named config constant, the
  TABLE_RERANK_MODEL discipline — mid-tier per ADR 032, supports `web_search_20260209`; the
  recorded alternative is `claude-sonnet-4-6`), `WEBSEARCH_MAX_USES = 3`,
  ⟨W2⟩ `WEBSEARCH_TIMEOUT_MS = 45_000` (must fit inside the raised 90s Server Action budget with
  the ~14s CBS pipeline max + margin — work item 7), ⟨W7⟩ `WEBSEARCH_MAX_TOKENS = 16_000`
  (sonnet-5 runs ADAPTIVE thinking by default and thinking tokens share the `max_tokens` budget —
  4096 risked routine truncation on 3-search rounds; 16K costs nothing extra unless generated),
  `interface WebSearchClient { search(question: string): Promise<WebSection> }`, and
  `AnthropicWebSearchClient` calling `sdk.messages.create` with the tool param from § Measured
  facts and the per-request timeout. Response handling: iterate
  `content`; a `web_search_tool_result` whose `content` is an error OBJECT ⇒ failed; collect
  `text` blocks carrying `citations` — each contiguous cited text block becomes one finding
  (text trimmed, hard-capped at 300 chars; citations mapped to `{url, title}`; only `http(s):`
  URLs kept); cap 4 findings; count `server_tool_use` blocks as `searches`. `stop_reason` not
  `end_turn` ⇒ failed `'stopped'` (includes `pause_turn` and `refusal` — v1 does not resume;
  recorded residual). Zero cited findings ⇒ `'no_findings'`. Any throw ⇒ `'api_error'`. The
  client NEVER throws.
- `prompt.ts`: `WEBSEARCH_PROMPT_VERSION = 1` + the Dutch system prompt (NEW bytes — allowed here
  only): search the web for the user's question; return at most 4 findings, each ONE short Dutch
  sentence stating only what a source says; no advice, no speculation beyond sources; prefer
  recent, authoritative sources; if nothing relevant is found, say so in one sentence without
  fabricating. Version recorded inside `WebSection.promptVersion` (NOT in the audit
  `PromptVersions` type — that shape stays untouched).
- `attach.ts`: 
  ```ts
  export interface WebBilling { reserve(): Promise<boolean> }  // injected closure; billing never imported
  export async function attachWebAugmentation(
    response: ComposedResponse,
    opts: { selection?: SourceSelection; client?: WebSearchClient; billing?: WebBilling },
  ): Promise<ComposedResponse>
  ```
  Always returns a NEW object with BOTH keys explicitly set (`sourceSelection: opts.selection ??

  attempt is OWED when: `selection?.web === true` AND `response.kind !== 'clarification'` AND

  `billing` is absent ⇒ `webSection = {status:'failed', code:'not_configured'}` (never a silent
  null); otherwise `await billing.reserve()` FIRST (false ⇒ `{status:'failed',
  code:'insufficient_balance'}` — debit-before-spend holds); then `client.search(question)`.
  When NOT owed: `webSection` is null (selection null, clarification, or skip-list). Whole body
  fail-soft (any throw ⇒ `'api_error'`); the CBS response is NEVER blocked. **The web section
  text is never concatenated into `response.text`** — pinned by test.
