## Invariants at stake (name these in every subagent brief)

- **R1/R8**: the fit gate reads METADATA only (`fetchTableSchema`), never observation cells; the
  delivered number still comes exclusively from ingested, validated cells via the audited pipeline.
- **Principle (b)**: all new CBS calls live inside the async job — the synchronous turn gains none.
- **Principle (c)**: every uncertain outcome falls toward next-candidate → honest refund; never a
  wrong table served confidently; the product never asserts a fact it did not verify (ADR 027 D2b:
  all-errored → `failed` with the honest infra message, never "geen passende maat").
- **Money**: the ledger primitives (`debitOnboarding`, `compensate`) and `refundOnboarding` stay
  byte-identical; debit amount, idempotency keys and refund semantics unchanged.
  `triggerOnboarding`/`createPendingRequest` widen only to CARRY `candidate_ids`.
- **Identity**: `pending_table_requests.table_id` is NEVER mutated (ADR 027 D2a) — the fit choice
  lives in the new `resolved_table_id`; the `(user_id, table_id)` active-unique index and the
  asking-twice dedupe are untouched by construction.
- **Prompt-byte discipline**: the intent parser's prompt is UNTOUCHED. The rerank prompt bytes
  change (forces a FULL tablefinder fixture re-record — the documentation constant
  `RERANK_PROMPT_VERSION` bumps to 2 but the re-record is forced by the bytes, not the constant).
  Do NOT touch `RERANK_SCHEMA_VERSION` or the prompt's "version is altijd 1" line — that is the
  OUTPUT-schema literal; changing one side breaks every rerank (validated by z.literal).

## Stage C — the fit gate in the job (hermetic)

New module `src/ingestion/onboarding-fit.ts` (fit prompt + schema + validator + version constant,
mirroring `rerank-*.ts`), driven from `processOneRow`:

```
if row.resolved_table_id is set:            // reclaimed retry after an accepted fit — deterministic
    resume at ingest with resolved_table_id  // resume, never re-run the fit loop (ADR 027 D2a)
else if row.candidate_ids is empty:          // legacy row → EXACTLY today's path, no fit gate,
    proceed with row.table_id as today       // no schema fetch, no fit LLM (ADR 027 D2c)
else:
    verdicts = []
    for candidate of row.candidate_ids:
        schema = source.fetchTableSchema(candidate)          // metadata only — on throw: record 'errored', next
        fit    = measureFit(row.question_text, schema)        // Haiku, closed choice + 'geen', hard allowlist
                                                              // on throw/invalid: record 'errored', next
        if fit accepts (>= calibrated threshold):
            setResolvedTable(row, candidate)                  // store helper: DB row AND in-memory object
            break
        verdicts.push('geen')
    if none accepted:
        if every candidate 'errored' → failAndRefund("Onverwachte fout …")        // honest infra failure
        else → unanswerableAndRefund("geen van de onderzochte tabellen bevat een maat die deze vraag beantwoordt")
// then the existing steps, unchanged, all reading (resolved_table_id ?? table_id):
// piggyback check → slice estimate → register+sync → vocab → delivery (gate intact)
```

- The fit result (measure code + one-line reading) is recorded on the row (`fit_note text`, same
  migration — diagnostics only; delivery does NOT consume it: the parser + delivery gate still
  decide independently, defense in depth).
- Tests (fixture-driven, `FixtureSource` with the Record<tableId, docs> form serving BOTH bijstand
  tables from Stage A's captures): candidate 1 misfits → candidate 2 delivers; all-'geen' →
  unanswerable with the scoped message; all-errored → failed with the infra message; fit-error on
  1 then accept on 2; legacy row (`[]`) → byte-identical today-path (no schema fetch — pin with a
  throwing-source guard); reclaim-after-accept resumes at ingest (no second fit loop, no second
  table ingested); re-ask during running → `alreadyPending`, no second debit (dedupe pin on the
  UNCHANGED table_id).
