# #154 design — per-cell freshness truth for retained cells (the false "gesynchroniseerd op" date)

**Written by the Fable overnight design marathon (2026-07-18→19), phase 4.** The s47 finding (CONFIRMED ×2, hunt 3):
under the #32 keep-and-count policy a cell CBS stops publishing is retained, but `cbs_tables.last_sync_at` bumps
unconditionally on every successful sync and the answer path attaches that TABLE-level timestamp to EVERY served cell
(`attribution.syncedAt` → *"Gegevens gesynchroniseerd op {date}"*), which `staleness.ts` also uses for its age check.
A retained cell therefore shows as freshly-reconfirmed forever and the staleness net never fires — a direct dent in
the public claim (*"met bron en datum getoond"* / CLAUDE.md public-claim rule). The value itself stays a real CBS cell
(principle (a) holds); this is a DATE-honesty fix. Exposure ≈ nil today (stable headline tables; CBS usually changes
values — caught by the corrections diff — rather than retracting cells), which is why this is a small, precise WP and
not an emergency.

## 1. The truth model — three facts, one honest column to add

Per cell there are three distinct freshness facts; today's schema stores (1) and (3) and the bug conflates them:

1. **Last-changed** — when the value/status last actually changed. Stored correctly: `observations.batch_id` (the
   `IS DISTINCT FROM` upsert guard leaves unchanged rows untouched). This is what the s47 finder's sketch wanted to
   display — **measured flawed**: an unchanged-but-reconfirmed cell keeps its OLD batch_id, so the sketch mis-dates
   the common case in the opposite direction (fresh cells shown stale). The rejection stands.
2. **Last-confirmed** — the most recent sync in which CBS still returned this cell (changed or not). **Does not exist
   in the schema. This is the honest "gesynchroniseerd op" for a cell.**
3. **Table last-sync** — `cbs_tables.last_sync_at`. Honest at table granularity; wrong as a per-cell claim only for
   the retained minority.

**Key observation that shapes the design:** for every cell that IS present in the latest successful sync, facts (2)
and (3) coincide — the sync reconfirmed it. They diverge only for the rare retained cells (`rows_missing` per batch).
So the schema does not need a per-cell timestamp maintained on every sync; it needs to mark the EXCEPTION.

## 2. Decision: track absence, not presence — a NULL-sentinel `last_seen_batch_id`

Migration **`021_observation_last_seen.sql`** (file-only until the supervised apply — 016/017/019/020 precedent):

```sql
alter table observations
  add column last_seen_batch_id bigint references ingestion_batches(id);
-- NULL (the default, and the backfill state for every existing row) means:
-- "present in this table's most recent successful sync" — the effective
-- synced-at date is cbs_tables.last_sync_at, exactly today's behavior.
-- NON-NULL means: "retained cell — CBS stopped returning it; last provably
-- confirmed in this batch" — the effective date is that batch's finished_at.
```

**Sync-side rules (all in `pipeline.ts`, same transaction as the upsert):**

- **Mark on absence (rare rows only):** the existing `NOT EXISTS` anti-join that counts `rows_missing` additionally
  sets `last_seen_batch_id` on those rows **only where it is still NULL** (never overwrite — repeated absence must
  not creep the date forward), to the **latest prior batch whose presence is provable**: the previous succeeded batch
  for this table if that batch ran after migration 021 was applied (the NULL-means-present invariant holds from then
  on), else the cell's own `batch_id` (its last change — a provably-true lower bound; "gesynchroniseerd op" may
  understate, never overstate). One transition-window approximation, explicit and conservative.
- **Clear on reappearance:** the upsert's update condition gains one term —
  `... where (value, status, value_attribute) is distinct from (excluded...) OR observations.last_seen_batch_id is
  not null` — with `last_seen_batch_id = null` in the `DO UPDATE` set. A retained cell that CBS re-publishes with an
  identical value would otherwise be blocked by the guard and stay falsely marked. Write cost stays ∝ anomalies.
- **Write amplification statement (why not "update every present cell every sync"):** stamping presence per-cell per
  sync rewrites the full table every batch (136k-row syncs measured in s54) — WAL, bloat, and sync-time cost for a
  fact the table-level date already carries for the 99%-case. Rejected; the sentinel keeps writes proportional to
  retractions and reappearances, which the #32 tolerance already bounds.

## 3. Answer path: effective date = the oldest cell truth (the aggregation decision)

`src/query/run.ts` (where `attribution` is built): the effective per-cell date is
`last_seen_batch_id IS NULL → table.lastSyncAt`, else that batch's `finished_at` (one extra query only when a result
actually contains retained cells). The result's single displayed `attribution.syncedAt` becomes the **minimum**
(oldest) of its cells' effective dates: with zero retained cells this is byte-identical to today; with any retained
cell the shown date honestly reflects the weakest link, and — because `checkStaleness` reads the same field — the
staleness warning starts firing on exactly the class that today slips the net. No new UI surface, no per-cell date
list (rejected: clutter for a rare case; the min is the conservative claim the public promise needs).

**Wording refinement (build detail, Dutch copy decided at build):** when the min came from a retained cell, the
staleness warning's *"onze laatste synchronisatie was op {date}"* is subtly off (we DID sync; the cell wasn't
reconfirmed) — switch to *"een deel van deze cijfers is door CBS sinds {date} niet opnieuw bevestigd"* in that branch.
The attribution line format itself does not change shape (R8 re-derivation stays byte-exact from stored fields).

## 4. Interactions, invariants, alternatives

- **R4** (source + last-sync date shown) is the invariant being repaired; **the staleness row** (05-data-rules
  failure table, warn-and-serve decision #18) regains its teeth for retained cells; **R11** untouched (provisional
  marking is orthogonal — no new per-cell prose marking in v1; a persistent-retraction operator ALERT stays #157(c),
  deliberately out of scope).
- **R8:** `attribution.syncedAt` is stored on the audit row and re-derived byte-identically — unchanged mechanism,
  now fed an honest value. The reconstruction needs the same effective-date computation, which reads only stored
  columns (deploy-order note: reconstruction of PRE-fix rows still matches, because their stored attribution carries
  the value that was shown at the time — R8 verifies what was said, not what should have been said).
- **Corrections diff, #157(a)/(b), quarantine, ±20% policy:** all untouched. Source-neutral (ADR 030): the column
  lives on `observations`, no CBS-specific semantics.
- **Alternatives rejected:** (1) the batch_id display sketch — measured flawed (§1); (2) per-cell presence stamp
  every sync — write amplification (§2); (3) a separate presence-log table (audit-grade batch×cell bitmap) — the
  honest maximum, but heavy machinery for a rare class the sentinel covers; revisit if multi-source tables show
  frequent retractions.

## 5. Rollout + the test that pins the s47 repro

**Deploy-order-safe and flag-free by construction:** the migration backfills nothing but NULL (= today's behavior);
the code change is inert until a post-apply sync marks the first absent cell. Ship order: merge code + migration file
→ supervised window: apply 021 → next scheduled syncs start marking. No env flip.

**Tests (hermetic, PGlite + fixtures):**
1. **The s47 repro, pinned:** sync fixture with cells A+B → second sync omits B (within ±20%) → assert
   `B.last_seen_batch_id = batch-1`, A stays NULL; a result containing B shows `syncedAt = batch-1.finished_at`;
   clock-injected staleness check FIRES for B's result and stays silent for A's — the exact net that today never
   fires.
2. **No-retained-cells regression belt:** every existing attribution/staleness test byte-identical (the min degrades
   to `last_sync_at`).
3. **Reappearance:** third sync returns B unchanged → column cleared via the OR-term, date back to `last_sync_at`.
4. **No date-creep:** two consecutive absent syncs → the marked batch id does not move.
5. **Transition lower bound:** first post-migration sync with an already-absent cell → marked with the cell's own
   `batch_id`, never a later batch.
6. **R8:** a retained-cell answer's audit row reconstructs byte-identically (attribution + staleness branch).

**Effort estimate:** one focused session (migration file + ~3 pipeline touch-points + run.ts effective-date + tests);
zero LLM spend (fully deterministic surface); the apply is a normal supervised-window step. Fits as a rider on any
owner-present session — e.g. the same supervised window as a WP26 go-live or a release-day sync.
