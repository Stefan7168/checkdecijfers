# Data rules

The CBS data strategy and the anti-hallucination guardrails, written as **checkable invariants** — each rule states how a test or code review verifies it. These operationalize the three confirmed principles (see [CLAUDE.md](../CLAUDE.md)).

## Data access strategy (confirmed: bulk ingestion)

- Ingestion uses **CBS OData v4 / bulk channels** through the `CbsSource` adapter — never the 10,000-cell-capped v3 API, never live queries from the request path (ADR [003](decisions/003-cbs-access-layer.md)).
- Each table in scope is registered in `cbs_tables` with: CBS table ID, title, the **full dimension list** with pinned default ("totaal") coordinates, period grains with per-grain coverage and **semantics** (the registry knows that a population table's `2025JJ00` means *stand per 1 januari* while CPI's means *jaargemiddelde* — B13's growth-in-2024 pairing depends on this), ingested slice ranges when a table is partially loaded, units, expected update cadence, last sync time, schema fingerprint, version. When a table is sliced, refusal wording must distinguish "outside the loaded slice" from "not published by CBS".
- **Canonical defaults**: the alias list maps each everyday term to one canonical headline measure and preferred table ("werkloosheid" → the seasonally-adjusted headline series). Answers always state the chosen definition ("werkloosheid, seizoensgecorrigeerd") — a default is transparent, never a hidden guess.
- Ingestion is idempotent and batch-recorded (`ingestion_batches`): re-running a sync must not duplicate or alter observations. **Verify:** ingestion test runs the same sync twice, asserts identical row *content* (table checksum/diff), not just counts.

## Validation pipeline (ingestion side)

Order of checks per sync; any failure marks the table `needs_review` and excludes it from answering — failures are loud, never silent:

1. **Schema fingerprint** matches the registry (dimension names, measure codes, units). Mismatch = suspected CBS redesign.
2. **Row plausibility**: row count within a defined tolerance of the previous sync (**Assumption:** default ±20%, per-table override in the registry — tuned during Phase 0); no empty measures; no duplicate cells (the same measure + full coordinates fetched twice = corrupted/overlapping fetch, fails the batch). Null values carrying a CBS reason (`ValueAttribute`: confidential, impossible, not yet available) are **valid rows, not failures** — they ingest with their status so refusals can state the true reason. A null value *without* a CBS reason fails the batch.
3. **Period parsing**: every CBS period code (`2024JJ00`, `2025KW04`, …) converts to a typed period; unparseable codes fail the batch. Every observed period must also appear in the table's published period list **and carry a publication status** (Definitief / Voorlopig / NaderVoorlopig — the R11 feed); a missing status fails the batch, never defaults to definitive.
4. **Dimension mapping**: every dimension coordinate — region, but equally Geslacht, Leeftijd, seizoenscorrectie, bestedingscategorie — resolves in `dimension_labels`; unknown codes fail the batch (new municipality codes appear via reviewed mapping updates, not silently).
5. **Unit consistency**: units per measure unchanged vs. registry, or flagged.

**Verify:** ingestion test suite feeds a fixture with each corruption type (renamed dimension, bad period code, unknown dimension code, changed unit, implausible row count — a truncated sync or empty measure) and asserts the batch fails with the right reason and the table is excluded from answering.

**Loud includes the operator.** Phase 0's ingestion CLI fails with a non-zero exit and a plain-language summary the owner can read. From Phase 1 (scheduled syncs, real users): batch failure, a `needs_review` transition, a missed expected sync, and an app health-check failure each **notify the owner** via the notification seam ([04-architecture.md](04-architecture.md); channel choice: [open-questions.md](open-questions.md) #23). A quarantine only the database knows about is silent, not loud.

**Table onboarding rule** (Phase 1 manual growth and Phase 2–3 demand-driven onboarding alike): a table leaves `needs_review` only when it ships with 2–3 benchmark-style verification tasks — frozen-key entries added to the automated benchmark. "Mapping review" *means* passing those tasks; nothing else counts. **This is a structural/automated check, not a manual approval step** — no human click is required, as long as the frozen-key values themselves come from independent verification (structural existence + label-consistency checks, plus a cross-referenced source where available). See [open-questions.md](open-questions.md) #21 for the automated-pipeline design and its honest multi-candidate fallback for genuine definitional ambiguity. Review throughput/SLA for user-triggered fetches: [open-questions.md](open-questions.md) #24.

## Anti-hallucination invariants (answer side)

| # | Invariant | Verified by |
|---|---|---|
| **R1** | Every numeric value in an answer traces to a query-result ID recorded in the audit record. | Automated test: scan rendered answer for numeric tokens; each must appear in (or be a registered derivation of) the linked result set. Exemptions are **structural, never pattern-based**: attribution fields (table ID, sync date, license version) are matched positionally from the answer schema, and period labels are matched against the validated intent — so a wrong year in prose still fails. Runs on every benchmark task. |
| **R2** | The answer-phrasing prompt receives **only validated result objects and attribution metadata** — never raw table rows, never the ingested table. | Code review rule: exactly one module builds that prompt, typed to accept validated result objects (`ValidatedResult`) only. Test asserts the serialized prompt contains no fields outside the schema. |
| **R3** | Numbers in LLM output match result objects **verbatim** (formatting may localize, value may not change; rounding only via registered derivations). The phrasing prompt requires quantities in **digits**; Dutch number-words and scale forms ("zeventien miljoen", "een kwart", "verdubbeld") are rejected unless backed by a registered derivation. | Post-generation validator (blocking) that canonicalizes digit tokens, Dutch number/scale words (duizend/miljoen/miljard) and percent forms before comparing; mismatch or unbacked word-form → one regeneration, then fail closed to a template answer. Unit-tested with seeded mismatches including word-form fabrications. |
| **R4** | Every answer displays: CBS table ID(s), table title, our last-sync date, covered period. | Snapshot/UI test on the answer component; spec-level test that chart specs contain attribution (ADR [007](decisions/007-chart-spec-rendering.md)). |
| **R5** | Derived values (differences, rankings, growth) are computed by deterministic code, marked as derived, and list their source cells. | Derivations exist only as registered functions in the query module (code review); R1's scan accepts a numeric token only if a derivation record links it to source result IDs. A snapshot test asserts the visible derived-marking (CC BY section below) renders whenever the answer schema contains a derivation record. |
| **R6** | Chart specs are built by deterministic code from validated results; the renderer cannot compute or omit. | Real since WP8 (`tests/chart/` + the R6 block in `tests/invariants/`): specs are deterministic verbatim projections (every cell exactly once, values/order unchanged, nulls kept with reasons); the rendered SVG shows no numeric token that is not a spec string (same tokenizer R1/R3 use), renders one resultId-tagged marker per point with each value label bound to its point (`data-label-for`), breaks line paths at null gaps, keeps the x-axis chronological, and marker geometry is asserted to be an exact affine image of the values; B4/B8 line charts reproduce the frozen answer key hermetically in CI (ADR [014](decisions/014-chart-spec-v1-and-renderer.md)). |
| **R7** | The intent parser returns **ranked candidate intents** with confidence; more than one candidate above the cutoff, or a top candidate below threshold → clarification, never a best guess — for **user-facing ambiguity** (region, period, materially different definitions). **Registry-internal variant choice** (which technical reading of one everyday term) resolves to the canonical default instead, with the chosen definition stated in the answer (canonical-default policy above). **Assumption:** threshold values and the calibration procedure (against a labeled set of ambiguous questions) are implementation-time decisions — see [open-questions.md](open-questions.md) #19. | Schema validation at the call site; threshold behavior tested against the labeled ambiguous-question set; benchmark tasks B15–B16 as regression. |
| **R8** | Every answer writes an audit record — including the **final rendered answer text and the emitted chart spec** — before being shown. | Real since WP10 (`tests/audit/` + the R8 block in `tests/invariants/`, ADR [016](decisions/016-audit-records.md)): the audited entry points return only after the `audit_answers` insert commits, and a failing audit store **withholds the answer** (fail-closed to the internal refusal — proven by test); `reconstructionReport` re-verifies each stored row **from the row alone** — the stored body re-passes the R3/R9/R10/R11 validator against the stored result (every numeric token maps to stored result IDs/derivations — R1's scan from the record), the attribution line re-derives byte-identically (positional, R4), final text and chart spec re-derive through the same deterministic builders — with tamper tests proving every check can fail. One row per refusal and clarification too, incl. the clarification-reply text + pending state and the three prompt-version constants (ADR 015 obligations). Benchmark scoring reads these records: `benchmark:run` dumps them, `benchmark:score` scores the dump against the frozen key in CI. |
| **R9** | Quantitative claims are **semantically bound** to the right data: each value sits with the dimension coordinates (region, period, and any other dimensions) it belongs to, and every direction/comparison/ranking statement ("gestegen", "meer dan", "de meeste") matches the validated results. The LLM may not introduce quantitative claims beyond the results. | Post-generation semantic check: for each matched number, adjacent entity/period tokens are checked against that result's dimension labels; direction/ranking words are checked against a registered derivation over the series (fail closed to a template answer). The query step **pre-registers direction and first/last derivations for every series result** so honest trend sentences have something to bind to, and the test suite includes correct-prose fixtures (legitimate paraphrases and direction words) asserting the validator passes them — false positives are bugs too. Benchmark rubric scores binding and direction correctness ([02-user-scenarios.md](02-user-scenarios.md), Scoring). |
| **R10** | The **unit** shown with each value matches the result object's unit metadata — guarding factor-1,000 misreadings ("443 euro" vs. "×1.000 euro") and %-vs-procentpunt confusion. | Answer schema carries a unit per value; the R3 validator checks the unit words adjacent to each matched number against that metadata. "Unit correct" is an explicit benchmark pass criterion. |
| **R11** | **Provisional figures are marked**: when a value's CBS status is not definitive (voorlopig / nader voorlopig), the answer and chart spec say so ("voorlopig cijfer"); null-with-reason cells state their CBS reason instead of appearing missing. | Status is a required field on result objects (observations schema, ADR [002](decisions/002-postgres-system-of-record.md)); snapshot test asserts the marking renders whenever status ≠ definitief; ingestion fixture covers null-with-attribute cells. |

## Failure behavior (confirmed principle: refuse, don't guess)

| Condition | Behavior | Regression test |
|---|---|---|
| Ambiguous intent (region/period/measure unresolved) | One compact clarifying question with concrete, *actually available* options; no numbers. "Actually available" is enforced, not assumed (WP15, [#56](open-questions.md)): a low-confidence echo suggestion is **dry-run through the real query layer** before being offered — an unservable one is replaced by a clarification naming what IS loaded, and the dry-run's return type carries no cell values by construction | B15, B16; tests/query/dry-run.test.ts; the #56 branch tests in tests/answer/intent-policy.test.ts |
| Follow-up question referencing the previous turn ("En in Rotterdam?") | Parsed **merged with the previous turn's resolved intent** (the stored query plan, ADR [016](decisions/016-audit-records.md)) offered as a structured, registry-vocabulary merge candidate (ADR [021](decisions/021-conversation-memory-structured-context.md)) — **never raw chat history in any prompt** (R2 surface, #41). A follow-up is a fresh question with its own single clarification round; every number still comes from the deterministic query layer | benchmark/followup-cases.json replayed hermetically (tests/answer/followup-parse.test.ts); context build/validate pins in tests/answer/context.test.ts |
| Newly onboarded table has genuinely multiple valid CBS definitions and no established canonical default (automated verification can't resolve which one the question means) | Show each candidate's real, correctly-attributed value with its own definition label — never silently pick one, never refuse outright | new fixture/benchmark task, TBD — tracked at [open-questions.md](open-questions.md) #21 |
| Requested data outside loaded scope | Refusal naming the scope limit + nearest answerable alternative | B17 |
| Question asks for prediction/opinion/causal interpretation | Refusal of the interpretation; offer underlying descriptive stats only if in scope | B18, B19 |
| Data exists but doesn't cover the requested period (stale/not yet published) | Freshness refusal: state freshest available period, offer it | B20 |
| Compound question (multiple independent asks in one message) | Honest split — "één vraag tegelijk; zal ik met X beginnen?" — not a misfired ambiguity clarification. Multi-ask decomposition is a Phase 1–2 roadmap item | fixture test with a two-ask question |
| Meta question about the product itself (bronnen, ontbrekende waarden, actualiteit, betrouwbaarheid, mogelijkheden) | Answered truthfully by a **deterministic product-behaviour template** (reason `meta`, ADR [022](decisions/022-meta-question-templates.md)) — routed by pattern **only inside the LLM's smalltalk classification**, so a data question can never be hijacked; zero LLM, no numeric token beyond whitelisted registry labels/example periods; unmatched smalltalk keeps the generic explanation | META_TEMPLATES structural sweep (routing, order-honesty, body-binding, no-numbers belt) in tests/answer/respond-refusals.test.ts; s-hallo + abandon-reply pins in tests/answer/respond-pipeline.test.ts |
| Still ambiguous after the one clarification round | Refusal-with-guidance (what to ask instead) — never a second question, never a silent default on a user-facing axis. The clarification itself must cover **all** unresolved axes at once (options are combined presets), and a free-text reply is parsed **merged with the pending partial intent**, not as a fresh question | B15/B16 post-clarification scoring; fixture with a free-text reply |
| Table marked `needs_review` (failed validation / suspected redesign) | Treated as out of scope; never served | ingestion fixture test |
| Table past its expected update cadence | Served **with an explicit staleness warning** if the requested period is covered; refused when the question implies recency (relative-time words such as "nu", "vandaag", "vorige maand", "meest recente"). **Decided** (Stefan, 2026-07-02): warn-and-serve for covered historical periods — [open-questions.md](open-questions.md) #18, resolved | clock-injected tests for **both** branches (warned-served and recency-refusal) |

## Source attribution & CC BY 4.0 obligations

CBS open data is licensed **CC BY 4.0**: attribution is required, and modified/derived data must be marked as such. Rules:

- **Attribution line on every answer and chart** (exact Dutch wording to be legal-checked, logged in [open-questions.md](open-questions.md)): *"Bron: CBS StatLine, tabel {ID} — {titel}. Gegevens gesynchroniseerd op {datum}. Licentie: CC BY 4.0."*
- **Derived-data marking**: any computed value carries *"bewerking van CBS-gegevens door checkdecijfers.nl"* alongside its source cells (pairs with R5).
- Attribution is part of the chart spec and answer schema, so it cannot be dropped by a rendering path (pairs with R4/R6).

## CBS platform-change risk

- **Table redesigns / mid-year schema changes** are expected, sometimes politically driven (definitions like "bijstand" change). Defense: schema fingerprints + `needs_review` quarantine (validation pipeline above); the registry keeps versions so an archived table's successor is an explicit re-mapping.
- **Silent retroactive corrections**: CBS corrects historical figures without announcement. Defense: syncs diff against previous values; changed historical cells are logged per batch (this diff is also the future scoop-alert seam). **Verify:** fixture test — a second sync with one changed historical cell must produce a batch log naming exactly that cell. This correction-diff log is also the **invalidation feed for published answer pages** (Phase 2): whether pages freeze as as-of snapshots with a correction banner or re-validate on sync is decided before Phase 2 design ([open-questions.md](open-questions.md) #22).
- **No fixed publication times**: cadence metadata is *expected* cadence, enforced as staleness warnings, not hard schedules.
- **Annual municipal reorganizations (gemeentelijke herindelingen)**: every 1 January, region codes change and CBS rolls the new indeling into regional tables — a *scheduled, knowable* event, not surprise corruption. Playbook: pre-load CBS's published new gemeente list as a reviewed `dimension_labels` update **before** the first January sync, and quarantine at the granularity of the new codes/periods rather than darkening a table's already-validated history. Cross-herindeling time series carry comparability breaks (trendbreuken) CBS marks in footnotes the pipeline doesn't ingest — herindeling awareness is a Phase 1 scope decision ([open-questions.md](open-questions.md) #26).
- **SDMX/.Stat migration** (announced; postponed indefinitely as of early 2026, not cancelled): the `CbsSource` adapter is the isolation seam (ADR [003](decisions/003-cbs-access-layer.md)). **Revisit trigger:** any CBS announcement re-activating the migration or deprecating OData → schedule a second adapter implementation.

## Audit-trail design

One `audit_answers` record per produced response — answer, refusal (with the refusal reason) **and clarification** (with its pending state; the reply round records the user's reply plus the pending clarification it answered — ADR [015](decisions/015-refusal-clarification-composition.md)/[016](decisions/016-audit-records.md)): question text, parsed intent (which **is** the query plan — the query layer is deterministic, so the executed query is a function of the stored intent + the pinned registry version, and the cells' result IDs + batch IDs are the execution references; ADR [016](decisions/016-audit-records.md)), result IDs and values, derivations applied, table IDs + versions + sync dates, model IDs and prompt versions used, timestamps, per-answer **latency and token counts** plus an **intent hash** (the measurement source for the caching/spend triggers in [06-roadmap.md](06-roadmap.md)), the **final rendered answer text**, and the **emitted chart spec** — making each record a complete, renderable snapshot (the seam the future shareable answer pages build on). Both the answer schema and the chart spec carry an explicit schema-version field (ADR [007](decisions/007-chart-spec-rendering.md)) — these records live forever.

Purpose now: benchmark scoring and R1/R8 verification (backend-only in Phase 0). Purpose later: the user-facing audit trail and premium exports (roadmap), and the single enforcement point for GDPR retention decisions ([04-architecture.md](04-architecture.md)).

**GDPR retention + self-service deletion (#14, built WP14 2026-07-05; onboarding legs added #120, 2026-07-12):** "these records live forever" above is now split by `source_tag` (WP13, [#44](open-questions.md)): **`benchmark`/`validation` rows still live forever** — they are this project's own regression fixtures, never a real user's personal data, and the purge below structurally excludes them. **Personal-data rows — `source_tag in ('user', 'onboarding_delivery')` — are retained 2 years** (the `onboarding_delivery` delivery answer carries the verbatim question + intent + answer, so it is a user's personal data just like a `user` row; added to the scope in session 39, [#120](open-questions.md)), and a signed-in user can delete their own question history on demand (owner decision, session 23) — both via `src/answer/audit/retention.ts`. The scope is a single shared `AUDIT_SCOPE` allowlist fragment in that module (never a `!= 'benchmark'` denylist), referenced by every scoped statement:

- **Retention purge** (`npm run gdpr:purge`, `scripts/gdpr-purge.ts`): redacts every `source_tag in ('user', 'onboarding_delivery')` audit row with `created_at` older than 2 years (`twoYearsBefore`). Dry-run by default (the preview counts come from `countPurgeableQuestionHistory`, built on the purge's OWN scope fragments so preview can never drift from `--apply`); `--apply` actually redacts. Idempotent — a second run against the same cutoff redacts nothing new.
- **Self-service deletion** (`web/app/actions.ts`'s `deleteMyQuestionHistory`, wired to a "Verwijder mijn vraaggeschiedenis" button in the account panel): a signed-in user's own `source_tag in ('user', 'onboarding_delivery')` rows, any age, scoped by the `getClaims()`-verified session id — never a client-supplied id.
- **Onboarding request text rides along (#120):** a question's free text is stored a SECOND time in `pending_table_requests` (migration 012's deliberate re-run-verbatim-at-delivery copy). Both paths therefore ALSO redact `pending_table_requests.question_text`/`topic_term`/`failure_summary` to the sentinel — self-service by the SAME bound `user_id`, purge by the SAME `created_at < cutoff` predicate — inside the SAME transaction as the audit-row redaction (a crash can never leave the question text behind in one store while it is gone from the other). ALL statuses are redacted, including `pending`/`running`: a GDPR erasure is not a job-state change. Money/status/id columns are untouched (the financial skeleton stays, mirroring the audit-row posture). Unlike the feedback delete, this leg runs UNguarded — `pending_table_requests` has existed since migration 012, so coupling it to another table's existence could only silently skip a real GDPR leg. **Accepted residual (in-flight race):** a redaction racing a `running` job doesn't stop it — its in-memory question copy still delivers, and the delivery audit row written AFTER the redaction survives until the next self-service click or purge (which now cover `onboarding_delivery`); a redacted `pending` row later claimed runs the sentinel through the pipeline → refuses → refunds (fail-safe, belt in `registerOnboardingVocabulary`); exposure today is nil (the owner is the only user). Recorded as a [#120](open-questions.md) residual.
- **Redaction, not physical deletion**: `credit_transactions.audit_answer_id` carries a plain FK to `audit_answers(id)` with no `ON DELETE` clause (migration 005 — deliberately not cascade). Any clarification/refusal row a compensation entry references would throw a foreign-key violation under a hard `DELETE`. Both the purge and the self-service action instead overwrite `question`/`final_text`/`response`/`reply_text`/`pending_clarification` with a fixed sentinel (`REDACTED_QUESTION_TEXT`, `"Deze vraag is verwijderd."`) **AND clear the promoted query-plan columns that reveal the topic — `intent`, `intent_hash`, `result_ids`, `table_ids`, `tables`, `conversation_context`** (owner decision, session 23: *"wis de inhoud volledig"* — a true "delete my question history" erases the subject/region/period too, not only the free text). The row, its id, and its ledger join all survive — only the skeleton (id, user_id, created_at, request_id, kind, token/latency metadata) remains, the minimal financial-trail record retained by law. This is also the exact match for the owner-decided UX: the dashboard renders a **"verwijderde vraag" placeholder row** (`web/components/question-history.tsx`) that keeps the credit amount visible and never hides the row.
- **Answer feedback rides along (WP128, #128):** the 👍/👎 store (`answer_feedback`, migration 017 — verdict + optional free text POINTING at an audit row, never modifying it) is user content too. Both paths **hard-DELETE** the matching feedback rows — self-service: all of the caller's feedback; purge: feedback attached to purged answers — inside the SAME transaction as the redaction, feedback first (a crash between the steps can never leave feedback text behind). Hard delete is safe here precisely because nothing references `answer_feedback` (no ledger FK), and it matches the owner's "wis de inhoud volledig" principle. Pins: `tests/audit/feedback.test.ts`.
- **The ledger (`credit_transactions`) is never written by either path** — both functions touch only `audit_answers` (+ the WP128 `answer_feedback` deletes), structurally (no import of `src/billing/ledger.ts` in `retention.ts`).
- Hermetic pins: `tests/audit/retention.test.ts` (scope-to-calling-user, ledger-untouched, purge-scoped-to-source-tag, purge-idempotent, the FK-violation-on-hard-delete proof, and the reviewer's *wis-de-inhoud-volledig* topic-scrub pin — subject/region/period do not survive in any promoted column).
