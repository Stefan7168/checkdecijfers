- [06-roadmap.md](06-roadmap.md) — the **phases** (0–3) at a high level.

## Sequencing note — why build order ≠ runtime order

The runtime pipeline runs intent → query → answer ([04-architecture.md](04-architecture.md)). The **build** order deliberately differs: we build the deterministic core (query / validation / derivations) **before** the LLM intent parser. Reasons (decided 2026-07-03): the deterministic layer is the anti-hallucination core, it needs no LLM or API key so it is fully testable in hermetic CI, and it lets us score B1–B14 against the frozen answer key with zero LLM involved — so when the parser lands next it targets a *known-good* query layer and a *fixed* intent contract, and any failure is unambiguously the parser's.

## WP6 — Intent parsing (LLM)  ✅ done 2026-07-03

Built as briefed; both open design questions resolved in ADR [012](decisions/012-intent-parsing-llm-harness.md): **hermetic CI** = record/replay fixtures keyed by a full-request hash behind an LLM-client seam (a changed prompt/schema/model/registry fails loudly; the live eval `npm run intent:eval` is off-gate), and **R7 thresholds** = 0.9/0.35, calibrated against a 45-case labelled set ([benchmark/intent-labelled-set.json](../benchmark/intent-labelled-set.json), resolves [open-questions #19](open-questions.md)). ## WP10 — Audit record per answer (R8)  ✅ done 2026-07-03

Built as briefed (ADR [016](decisions/016-audit-records.md) records the load-bearing choices): one `audit_answers` row per produced response (answers, refusals **and clarifications** — migration 004), written by audited wrap functions **before** the response is returned, with a **fail-closed policy** (a failing audit store withholds the answer/clarification — internal refusal — and annotates refusals); the envelope is stored verbatim as the authoritative snapshot with promoted scalar columns for querying; the stored `intent` is the query plan (deterministic layer); `reconstructionReport` re-verifies every row **from the row alone** (stored body re-passes the R3/R9/R10/R11 validator, attribution/text/chart re-derive byte-identically, tamper tests prove teeth). The ADR 015 wrap-site obligations are discharged: reply text + pending clarification on reply rows, the three prompt-version constants + per-call model IDs/token counts/latency on every row. Done-criterion met and in CI: R8 is a real invariant test (the suite carries **zero** todos — all eleven invariants tested), and the **benchmark scorer reads audit records**: `benchmark:run` drives all 20 tasks + the B15/B16 reply rounds + the B3/B5 un-disambiguated variants through the audited pipeline hermetically and dumps the rows; `benchmark:score` scores the dump against the frozen key and fails CI below the docs/03 gate ([STATUS.md](STATUS.md) for measured results).

## WP12 — Minimal chat UI + first deploy  ✅ done 2026-07-04

*Added 2026-07-03 (WP11 session): this entry closed a plan gap — [03-mvp-scope.md](03-mvp-scope.md) lists "Chat UI, minimal" as Phase 0 in-scope, and three earlier decisions deferred work "to the chat-UI session" — but the plan previously ended at WP11.*

Built as briefed (ADR [018](decisions/018-chat-ui-and-deploy.md) records the load-bearing choices, including a mid-session revision): `web/` is a Next.js App Router chat UI calling only the two audited entry points (`answerQuestionAudited`/`answerClarificationReplyAudited` via a thin Server Action, `web/app/actions.ts`); a Recharts wrapper (`web/components/chart.tsx`) renders ChartSpec v1 with the same honesty contract as the SVG renderer (formattedValue-only display, chronological period sort, honest null gaps, no invented axis labels); a `web/backend` symlink to `../src` reaches backend source with zero duplication. **Revised mid-session:** `web/` is a fully independent npm project (own `package-lock.json`), not an npm workspace member as originally planned — a shared workspace lockfile repeatedly fought Vercel's build assumptions, traced through six real CI deploy attempts before landing on the actual fix (a combination of the independent lockfile, the symlink, a Vercel project setting undocumented outside Vercel's own docs, and declaring the backend's runtime deps in both `package.json`s). CI's `gate` job gained `web/`'s own typecheck + 6-test suite; a `deploy` job (`needs: gate`) runs the Vercel CLI and is the only thing that ever deploys (no git integration connected) — deploy-blocking-on-red is live.

**Done-criterion met, measured against the real deployment (2026-07-04): all four `ComposedResponse` kinds work at https://checkdecijfers.vercel.app** — a golden-path answer with attribution, a chart-producing question (working tooltip, correct gaps/sort), a clarification round (reply resolved live to the frozen key's answer), and a direct scope refusal. *(Two of the four were initially measured only against a local production build while the docs claimed the deployment — caught same day by the post-session review's docs-honesty lens and fixed by measuring them live, not by weakening the claim.)* Root gate unaffected throughout: 445/445 tests, clean typecheck. A real backend bug was found and fixed along the way (`src/db/client.ts`'s CA-cert loading broke under Next's bundled runtime, invisible to every existing test). Full six-run deploy-debugging account, including two live-project operational mistakes (one caught by the harness's permission system before causing harm), in ADR 018 and [lessons-learned.md](lessons-learned.md). **The per-WP adversarial review ran post-session, owner-requested (WP12 shipped without it — a process gap recorded in lessons-learned): 14 findings confirmed incl. two HIGH (the WP8 membership-without-binding gap recurred in the new Recharts wrapper; an ADR-claimed test that did not exist), all fixed + test-pinned same day; scoreboard in [STATUS.md](STATUS.md).**
