**How this differs from the neighbours** (so nothing is duplicated):

## Sequencing note — why build order ≠ runtime order

The runtime pipeline runs intent → query → answer ([04-architecture.md](04-architecture.md)). The **build** order deliberately differs: we build the deterministic core (query / validation / derivations) **before** the LLM intent parser. Reasons (decided 2026-07-03): the deterministic layer is the anti-hallucination core, it needs no LLM or API key so it is fully testable in hermetic CI, and it lets us score B1–B14 against the frozen answer key with zero LLM involved — so when the parser lands next it targets a *known-good* query layer and a *fixed* intent contract, and any failure is unambiguously the parser's.

## WP19 — Dashboard polish: collapsed clarification rounds, live balance, low-balance warning, credits explainer

*The first of the three remaining Phase 1 adds in the session-19 owner-delegated order (dashboard polish → CSV export [#52] → anonymous trial [#53]). Four owner-decided items, no open product decisions left: [open-questions #67](open-questions.md) (collapse a clarification round into one history item), [#68](open-questions.md) (live balance updates), [#69](open-questions.md) (low-balance warning), [#76](open-questions.md) (credits-economy explainer under "Credits kopen"). Pure display/UX work: no migration, no prompt or schema change, zero LLM involvement, zero live spend.*

**#67 — the data piece (`src/billing/history.ts`, read-time only):** a clarify round is structurally at most two audit rows — the `kind='clarification'` row and one reply row (`reply_text`/`pending_clarification` set), because `respondToClarificationReply` converts any residual ambiguity into a `still_ambiguous` *refusal*, never a second clarification (verified in `src/answer/respond/respond.ts` before this brief). Both rows share the original question text (`question` on the reply row is `pending.question` — respond-audited.ts), and the reply row carries the answered `pending_clarification.questionNl` verbatim while the clarify row's envelope carries the same string at `response->'pending'->>'questionNl'` — so the pair is linkable at read time, walking rows oldest→newest and matching a reply to the nearest open clarification with the same (question, questionNl) signature. **No new column and no migration**: the link is reconstruction from data both rows already carry (the WP17 precedent inverted — that WP needed migration 010 because the ledger side genuinely lacked the key; here nothing is missing). **Known limitation (adversarial-review verdict):** this is a value-match over client-influenced fields, not a stored row link — two simultaneously-open byte-identical clarifications (same question in two tabs) can swap partners in the display, the second reply then standing alone; test-pinned as intended degradation, display-only, per-user, cost-safe (each row's cost joins the ledger on its own `request_id`). A guaranteed link needs a schema + `PendingClarification` change — deliberately deferred; revisit if real users hit it. Grouped entry: original question, the clarify row's `final_text` + the user's reply shown inside the fold, final outcome kind/text from the reply row, `creditsCharged` = the SUM of both turns' reconstructed net costs (null if either side is unattributable — never a partial sum presented as a total). Unmatched rows (an ignored clarification, a reply whose partner fell outside the fetch window) stay honest standalone entries. The fetch over-reads (2× the entry cap, a round being at most two rows) and caps *entries*, not rows.

**#68 + #69 + #76 — the web piece:** a new client boundary (`web/components/dashboard.tsx`) owns the displayed balance: `Chat` gains an `onOutcome` callback reporting each `GatedResponse`; an `'ok'` outcome decrements the displayed balance by the gate's own `netCost` (the #68 decision verbatim — no extra DB read), and an `insufficient_credits` outcome syncs the display to the server-reported balance it already carries — **a session-added extension beyond the #68 decision's letter** (which covers only the `ok` decrement), added because a stale higher number next to a "niet genoeg credits" message would be a visible lie; same no-extra-read principle, flagged in [open-questions #68](open-questions.md) for the owner to veto. `QuestionHistory` stays a Server Component, passed through as children. `AccountPanel` renders the #69 warning exactly per the owner threshold — visible when `simple ≤ balance < 2×simple`, hidden below (the `insufficient_credits` refusal owns that region, per the decision) — and the #76 explainer directly under the "Credits kopen" button: signup-grant size, the simple-question price, and the rough question count (`floor(grant / simple)`), one to two sentences, Dutch product copy. **Every number shown is read live from the pricing tables at render time** (`getActionClassPrice(db,'simple')` + a new `getSignupGrantCredits` read in `pricing-read.ts`) — never hardcoded in copy, per ADR [006](decisions/006-auth-billing-seams.md)'s prices-must-be-easy-to-change rule; the threshold therefore tracks a price change automatically.

**Invariants at stake:** the WP17 gate-consistency pin extends to rounds — history must reproduce the gate's own `netCost` arithmetic for a full clarify→reply round driven through the REAL `chargeAndRun` against the REAL pricing config, never hardcoded refund numbers; ADR 006 (no price literals in UI copy — including the #69 threshold and #76 explainer); the ledger stays append-only and this WP stays read-only against it; the hard module boundary (nothing in `src/answer/intent|compose`, `src/query`, `src/chart` changes); R8 untouched (no audit schema change); no fabricated numbers (every rendered figure traces to a DB read, the ledger join, or the gate's returned `netCost`).

**Done means:** full local gate green (all nine backend suites, hermetic benchmark run/score pair, web tests, clean typecheck both sides, real `next build`); new tests covering the round grouping (gate-consistency, unmatched-row honesty, mixed-null cost, entry-cap semantics), the balance decrement + insufficient-credits sync, the warning's exact boundary behaviour at `simple−1 / simple / 2×simple−1 / 2×simple`, and the explainer bound to config-driven props; the adversarial multi-lens review per the house rule; docs updated (#67/#68/#69/#76 marked built, STATUS.md session entry); CI green on the push.
