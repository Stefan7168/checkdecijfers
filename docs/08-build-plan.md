**How this differs from the neighbours** (so nothing is duplicated):

## Sequencing note — why build order ≠ runtime order

The runtime pipeline runs intent → query → answer ([04-architecture.md](04-architecture.md)). The **build** order deliberately differs: we build the deterministic core (query / validation / derivations) **before** the LLM intent parser. Reasons (decided 2026-07-03): the deterministic layer is the anti-hallucination core, it needs no LLM or API key so it is fully testable in hermetic CI, and it lets us score B1–B14 against the frozen answer key with zero LLM involved — so when the parser lands next it targets a *known-good* query layer and a *fixed* intent contract, and any failure is unambiguously the parser's.

**Format decision — CSV in Dutch notation** (`;` separator, decimal comma, no thousands grouping, UTF-8 BOM, CRLF): matches CBS StatLine's own Dutch-notation CSV download convention and opens correctly on double-click in Dutch-locale Excel — the audience is Dutch journalists. The BOM is required for Excel to detect UTF-8 (CBS titles carry accented characters).

**Numbers: verbatim AND machine-parseable.** The `waarde` column deliberately does **not** use the shared `formatValueNl` — that formatter groups thousands ("18.044.027") for prose display, which breaks naive numeric parsing outside Dutch-locale Excel, and CBS's own CSVs don't group. **Attribution travels INSIDE the file** (R4 + CC BY, per the overnight brief): the first preamble row is `Bron;` + **the verbatim output of `buildAttributionLine(result)`** — the same single builder the answer text and chart specs use, so the exported attribution structurally cannot drift (one-builder discipline).
